# slc-data-stage

## Use this skill when
- the task is to create or edit an `@tdb/slc-data` stage profile
- the task is to run a stage profile and produce staged output
- the task is to inspect or verify staged output for an `@tdb/slc-data` mount
- the task is to understand how the published CLI drives profile-based staging

## Do not use this skill when
- the task is primarily about deploying already-staged output to a remote target
- the task is primarily about editing lower-level pipeline internals rather than using the CLI workflow

## Boundary
- This skill is the orchestration layer. The published CLI is the execution layer:
  - `deno run -A jsr:@tdb/slc-data/cli <command> --flags`
- The local task:
  - `deno task cli ...`
  is a convenience wrapper over the same CLI surface.
- Never import `SlcDataCli` or call pipeline internals programmatically.
- Never reimplement schema validation, path resolution, mount refresh, or content hashing.
- If the CLI cannot do what is needed, fix the CLI first.

## Authority
Primary (works from outside the source tree):
- the published CLI itself:
  - `deno run -A jsr:@tdb/slc-data/cli --help`
- this package's `./deno.json` task surface

Source checkout only (use only when you are inside the package source and need to debug the CLI itself):
- `./-scripts/task.cli.ts` → local CLI task entry
- `./src/fs/m.cli/` → CLI implementation
- `./src/fs/m.DataPipeline/` → staging pipeline internals

Deeper authority (source checkout only, when understanding or repairing the pipeline boundary):
- `./src/fs/m.DataPipeline/m.stageFolder.ts` → end-to-end staging flow
- `./src/fs/m.DataPipeline/u.mounts.ts` → staged root mount index generation
- `@sys/model-slug` surfaces used by the pipeline:
  - `SlugTreeFs`
  - `SlugBundle.Transform.TreeFs`

## Workflow

### 1. Inspect the CLI surface
```
deno run -A jsr:@tdb/slc-data/cli --help
```
If this returns an error or empty output, and you are inside the source checkout, fall back to reading `./src/fs/m.cli/u.help.ts` and `./src/fs/m.cli/mod.ts`.

### 2. Create or edit a profile
```
deno run -A jsr:@tdb/slc-data/cli create --profile <name> --source <path>
```
Or edit the YAML file directly (see "Profile contract" below).

### 3. Run the profile
```
deno run -A jsr:@tdb/slc-data/cli stage --profile <name>
```

### 4. Verify staged output
Inspect the output tree (see "Verification" below).

## Profile contract

Location: `./-config/@tdb.slc-data/stage/<profile>.yaml`

Shape (exactly two fields, no additional properties):
```yaml
source: <path>    # absolute or relative to cwd
mount: <id>       # matches: ^[a-zA-Z0-9]+([._-][a-zA-Z0-9]+)*$
```

- Prefer `deno run -A jsr:@tdb/slc-data/cli create ...` over hand-authoring when creating new profiles.
- When editing an existing profile, read the file first and preserve its intent.
- The default mount name is `sample-1`.

## Verification

Verification is observational, not computational. After staging, confirm the output tree exists and has the expected shape.

Staging root: `./.tmp/staging.slc-data/`

Per-mount output at `./.tmp/staging.slc-data/<mount>/`:
```
<mount>/
  manifests/
    slug-tree.<mount>.json          # tree manifest
    slug-tree.<mount>.yaml          # tree manifest (YAML)
    slug-tree.<mount>.assets.json   # content index
  content/
    <sha256>.json                   # individual file content docs
```

Root index at `./.tmp/staging.slc-data/mounts.json`:
- Auto-generated by the CLI after each staging run.
- Lists all mounts currently present under the staging root.
- Check that the staged mount appears in this file.

Checks:
- `./.tmp/staging.slc-data/<mount>/` directory exists
- `manifests/slug-tree.<mount>.json` is present and non-empty
- `content/` contains at least one `.json` file
- `./.tmp/staging.slc-data/mounts.json` includes the mount name

Do not recompute hashes, re-derive manifests, or re-validate schemas unless debugging the CLI itself.

## Notes
- Always use non-interactive CLI subcommands (`create`, `stage`). The interactive menu exists but cannot be driven by an agent.
- If the CLI is broken, fix the CLI first. Do not build workflow assumptions on a broken execution layer.
- Use the deeper authority references only to understand or repair the staging pipeline. Do not bypass the CLI in normal workflow use.
- Deploy is downstream and out of scope for this skill.
