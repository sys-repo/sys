# Plan: @sys/driver-pi sandbox remove tool

## Status
Implemented in `@sys/driver-pi`: profile-gated `tools.remove` schema, wrapper-owned sandbox FS extension materialization, launcher prompt truth, run integration, and focused tests/checks.

## Problem
The current Pi agent tool plane exposes `read`, `bash`, `edit`, and `write`, but no deletion primitive.

The @sys system prompt correctly forbids using `bash` as a file-content or filesystem mutation escape hatch. That preserves containment, but it creates a real cleanup gap:

- refactors can create replacement files
- old files cannot be removed through `read`/`edit`/`write`
- the agent is forced to ask the human to delete stale files manually

The fix must preserve the containment model. It must not turn `bash` back into an ambient filesystem mutation tool.

## Decision

Add a wrapper-owned Pi extension that registers a single custom tool named `remove`.

Profile policy:

```yaml
tools:
  remove:
    enabled: true
    recursive: true
```

Generated extension path:

```text
.pi/@sys/extensions/sandbox.fs.ts
```

Launch model:

```text
Pi child args include:
  --no-extensions
  --extension .pi/@sys/extensions/sandbox.fs.ts
```

This keeps ambient global/project Pi extensions disabled while explicitly loading only the wrapper-owned extension. Pi's resource loader still permits CLI-specified extensions when `--no-extensions` is set; `--no-extensions` suppresses ambient discovery, not explicit CLI extension paths.

## Naming decision

Use `remove`, not `remove_path`, `delete`, `rm`, or `unlink`.

Rationale:

- `remove` matches the existing primitive verb style: `read`, `edit`, `write`.
- The schema already makes the object a filesystem path; repeating `path` in the tool name is noise.
- `delete` is broader and less filesystem-specific.
- `rm` imports shell semantics into a non-shell tool.
- `unlink` is Unix-specific and file-only.
- `remove-path` is nicer in YAML but risky as an LLM/provider function name.
- `remove_path` is provider-safe but ugly and over-specified.

The YAML key and Pi tool name should both be `remove`.

## Authority model

This design has three separate lanes that must not be collapsed:

1. **Process permissions** — Deno `--allow-write` grants needed for Pi to run.
2. **Remove operation roots** — paths the `remove` tool is allowed to target.
3. **Prompt truth** — the model-visible description of available tools and rules.

The remove operation roots are intentionally narrower than process write permissions. Runtime, cache, home, and temp paths may be writable by the Pi process but must not become removable merely because the process needs them.

## Core invariants

1. Bash stays constrained.
   - Do not relax prompt rules for `bash`.
   - Do not bless `rm`, `unlink`, shell redirection, or ad hoc scripts.

2. The Deno sandbox remains the hard authority.
   - `tools.remove.enabled` does not add filesystem permission.
   - `sandbox.capability.write` still defines process write permission.
   - Deno permission denial is a valid and expected safety boundary.

3. Remove operation roots stay narrower than process write roots.
   - Derive operation roots before `resolveWrite(...)` adds runtime/temp write grants.
   - Operation roots are the runtime root plus explicit profile/caller write roots.
   - Process-only runtime, cache, home, and temp roots are not remove roots unless explicitly granted as such.

4. The extension is wrapper-owned.
   - It is generated by `@sys/driver-pi`.
   - It is not discovered from user/global extension directories.
   - Ambient extensions remain disabled by default.

5. The tool is narrow.
   - One path input.
   - No globs.
   - No shell command strings.
   - No batch delete in the first version.
   - No implicit recursive removal.

6. Recursive removal is a separate policy bit.
   - Tool call `recursive: true` is refused unless profile policy has `tools.remove.recursive: true`.
   - Default/backfilled `recursive` is `true` while `enabled` remains `false` until explicit opt-in.

7. Path containment is concrete, not vibes.
   - Reject empty paths.
   - Reject raw `..` segments.
   - Reject glob-shaped inputs.
   - Resolve lexical containment against operation roots.
   - Reject intermediate symlink traversal.
   - Allow final symlink removal only as non-recursive removal of the symlink itself.

8. Control-plane paths are protected.
   - Refuse removal of `.git` and descendants.
   - Refuse removal of `.pi` and descendants.
   - Refuse removal of wrapper runtime/log/temp descendants even when they sit under an operation root.
   - Refuse removal of operation roots themselves.
   - Remove specific child paths, not whole control/runtime roots.

9. The prompt stays truthful.
   - When `remove` is enabled, the model-visible prompt must mention it.
   - Do not rely on Pi extension `promptSnippet`/`promptGuidelines` alone because the wrapper supplies a custom `--system-prompt`.
   - The same resolved policy that materializes the extension must drive the prompt contract.

10. Errors must be useful.
   - Permission errors should name the path and the missing write-scope implication.
   - Policy errors should say which profile policy blocked the operation.
   - Missing files should be reported as missing, not treated as success in v1.

## Tool contract

Pi tool name:

```ts
remove
```

Tool parameters:

```ts
type RemoveInput = {
  readonly path: string;
  readonly recursive?: boolean;
};
```

Semantics:

- `path` must be non-empty after trimming.
- `path` is literal; glob characters are rejected rather than expanded.
- `~` is not expanded; leading `~` is rejected to avoid false home semantics.
- raw `..` path segments are rejected for both relative and absolute inputs.
- relative paths resolve against `ctx.cwd`.
- absolute paths are accepted only when normalized inside a remove operation root.
- non-recursive removal may remove a file, an empty directory, or a final symlink itself.
- recursive removal is only for directory trees and requires both:
  - input `recursive: true`
  - profile `tools.remove.recursive: true`
- intermediate symlink traversal is rejected before removal.
- final symlink removal is allowed only when `recursive` is false.
- removal is performed with the Deno runtime remove primitive, not shell.

Result shape:

```ts
type RemoveResult = {
  readonly ok: boolean;
  readonly path: string;
  readonly resolved: string;
  readonly recursive: boolean;
  readonly reason?: string;
};
```

The returned Pi tool result should encode success or error in the normal `AgentToolResult` shape:

- success: `content: [{ type: 'text', text: 'Removed: ...' }]`
- failure: `isError: true`, with a terse reason
- details: structured `RemoveResult`

## Profile schema changes

Extend `PiCliProfiles.Yaml.Profile`:

```ts
export type Profile = {
  readonly prompt?: PiCliProfiles.Prompt;
  readonly sandbox?: PiCliProfiles.Sandbox;
  readonly tools?: PiCliProfiles.Tools;
};
```

Add:

```ts
export type Tools = {
  readonly remove?: Tools.Remove;
};

export namespace Tools {
  export type Remove = {
    readonly enabled?: boolean;
    readonly recursive?: boolean;
  };
}
```

Schema policy:

- `additionalProperties: false` at every level.
- absent `tools.remove.enabled` resolves to disabled.
- generated default profiles should include the stanza explicitly as disabled for discoverability.
- runtime must not silently enable `remove` for older profiles.

Reason for absent-means-disabled:

- avoids adding a new destructive operation to existing profiles by surprise
- makes the first enablement an explicit profile decision
- keeps the tool surface auditable by reading the profile

## Template/materialization design

Use the canonical FileMap + `@sys/tmpl-engine` pattern, following the shape used in `@sys/cell`.

Proposed module:

```text
src/m.core/m.extension/
  mod.ts
  t.ts
  common.ts
  m.sandbox.fs/
    mod.ts
    t.ts
    common.ts
    -bundle/
      mod.ts
      -bundle.ts
      -bundle.json
    tmpl.sandbox.fs/
      sandbox.fs.ts
    u.make.ts
    u.write.ts
```

The exact landing should be scaffolded through the canonical template flow before implementation.

Responsibilities:

- `m.extension` owns wrapper-supplied Pi extensions.
- `m.sandbox.fs` owns the first extension: sandbox filesystem tools.
- `u.write.ts` materializes the generated extension into `.pi/@sys/extensions/sandbox.fs.ts`.
- `u.make.ts` consumes the embedded FileMap and injects launch-specific policy.

Bundle flow:

```text
tmpl.sandbox.fs/*
  -> TmplEngine.bundle(...)
  -> -bundle/-bundle.json
  -> imported as FileMap
  -> TmplEngine.makeTmpl(fileMap, processFile)
  -> write(root/.pi/@sys/extensions/sandbox.fs.ts)
```

Processing rules:

- assert generated target paths are relative
- reject `..` and absolute template paths
- rename template root path to `sandbox.fs.ts`
- inject a serialized policy object containing:
  - `recursive`
  - remove operation roots, not process write roots
  - protected roots and descendants
- derive protected roots from wrapper-owned runtime paths, including `.git`, `.pi`, and any legacy runtime/log/temp roots still supported by the launcher
- ensure generated files contain no unresolved template markers

Use `@sys/fs` and `@sys/tmpl-engine` through local `common.ts` barrels in driver-pi code.

The generated Pi extension itself is an interop artifact. It may use the upstream-supported Pi extension imports (`@mariozechner/pi-coding-agent`, `typebox`) and the Deno runtime remove primitive because this wrapper launches Pi under Deno.

## Runtime integration

Current profile run flow:

```text
m.cli.profiles.run
  -> resolveRun(...)
  -> m.cli.run(...)
  -> PiArgs.toArgs(...)
  -> deno run ... pi
```

Add extension materialization during profile resolution, after profile YAML has been validated and profile/caller write roots have been resolved.

Desired flow:

```text
resolveRun(...)
  -> read/validate profile
  -> resolve prompt/context
  -> resolve profile/caller write roots
  -> derive remove operation roots from runtime root + explicit profile/caller write roots
  -> derive process write roots separately for the Pi process
  -> if tools.remove.enabled:
       write sandbox.fs extension under runtime root
       append launcher-owned remove prompt contract
       append ['--extension', extensionPath] to Pi passthrough args
  -> return resolved run args
```

`PiArgs.toArgs(...)` can continue to emit `--no-extensions`; explicit `--extension` args should appear in the passthrough args after the disable flags. Pi's resource loader allows CLI-specified extensions through this lane while still suppressing ambient extension discovery.

Important: do not write extension paths into `.pi/agent/settings.json`. This is a wrapper launch decision, not a persistent user Pi setting.

### Prompt truth contract

Because `@sys/driver-pi` launches Pi with a custom `--system-prompt`, Pi's default prompt builder does not automatically include extension `promptSnippet` or `promptGuidelines` text. Therefore the wrapper must own prompt truth for wrapper-owned tools.

When `tools.remove.enabled` is true:

- append a launcher-owned `--append-system-prompt` section that names `remove`
- state that `remove` is the cleanup primitive for stale files/directories
- state that `bash` must not be used for deletion/cleanup
- state whether recursive removal is enabled or disabled
- derive this text from the same resolved policy used to generate the extension

When `prompt.system` is omitted or `null`, the wrapper-owned default prompt must not contain an exhaustive tool list that omits `remove`. Prefer a helper that renders a complete active-tool contract, or rename the static list as base tools and rely on the runtime tool contract annex for additions.

When `prompt.system` is an explicit replacement string, still append the launcher-owned runtime tool contract. A custom prompt replaces style and base instruction prose; it does not replace trusted launcher runtime facts.

## Generated extension behavior

High-level pseudocode:

```ts
import type { ExtensionAPI } from '@mariozechner/pi-coding-agent';
import { resolve } from 'node:path';
import { Type } from 'typebox';

const POLICY = {
  recursive: true,
  removeRoots: [...],
  protectedRoots: [...],
} as const;

export default function sandboxFs(pi: ExtensionAPI) {
  pi.registerTool({
    name: 'remove',
    label: 'Remove',
    description: 'Remove a file or directory path inside the writable sandbox. No globs, no shell commands.',
    promptSnippet: 'Remove a file or directory path inside the writable sandbox.',
    promptGuidelines: [
      'Use remove only for stale files or directories that should no longer exist after a refactor.',
      'Do not use bash for file deletion; use remove for cleanup inside the writable sandbox.',
      'Set recursive only when removing a directory tree and profile policy permits recursive removal.',
    ],
    parameters: Type.Object({
      path: Type.String({ description: 'File or directory path to remove, relative to cwd or absolute inside the writable sandbox.' }),
      recursive: Type.Optional(Type.Boolean({ description: 'Remove a directory tree. Requires profile policy tools.remove.recursive.' })),
    }),
    async execute(_toolCallId, params, _signal, _onUpdate, ctx) {
      const requested = params.path.trim();
      const recursive = params.recursive === true;
      const target = resolve(ctx.cwd, requested);

      const guard = await guardRemove({ requested, target, recursive, policy: POLICY });
      if (!guard.ok) return toError(requested, target, recursive, guard.reason);

      await deno().remove(target, { recursive });
      return toSuccess(requested, target, recursive);
    },
  });
}
```

`guardRemove` owns all policy checks: raw path shape, operation-root containment, protected roots and descendants, existence, directory kind, and symlink traversal. `Deno.remove` is called only after the guard returns success.

Use a small runtime accessor for Deno so failure is explicit if the extension is ever run outside the wrapper-owned Deno launch path. The accessor should read `globalThis.Deno` and carry a minimal local type for `lstat`/`remove`; do not depend on ambient Deno types inside the Pi extension. Do not use Node filesystem removal APIs or shell execution in the generated extension.

## Security review

### Threat: prompt asks for `rm -rf`

Mitigation:

- prompt still forbids bash cleanup
- custom tool has no shell surface
- recursive removal goes through the dedicated protected `remove` tool, not `bash`
- launcher-owned prompt contract says to use `remove` instead of bash for cleanup

### Threat: prompt lies by omission

Mitigation:

- do not rely on Pi extension prompt snippets because custom `--system-prompt` bypasses them
- append a launcher-owned runtime tool contract whenever `remove` is enabled
- tests assert the enabled tool is visible in resolved prompt args

### Threat: remove repo root or operation root

Mitigation:

- generated policy blocks exact operation roots
- `.git`, `.pi`, and wrapper runtime descendants are protected
- recursive removal defaults true only after explicit `remove` opt-in; operation roots still cannot be removed

### Threat: remove process-only runtime/temp paths

Mitigation:

- operation roots are derived separately from process write roots
- generated policy does not include temp/cache/home roots added for process execution
- tests assert process-only roots are not present in generated remove policy

### Threat: remove path outside sandbox

Mitigation:

- generated extension performs lexical operation-root guard
- generated extension rejects raw `..` segments and glob-shaped inputs
- Deno `--allow-write` remains hard authority

### Threat: symlink traversal escapes operation roots

Mitigation:

- reject intermediate symlink traversal before remove
- allow final symlink removal only as non-recursive symlink removal
- reject recursive removal of final symlinks

### Threat: stale generated extension persists

Mitigation:

- extension is rewritten on each enabled launch
- generated path is under `.pi/@sys/extensions/`
- Pi is launched with explicit `--extension` path
- if policy disabled, do not pass the extension path

### Threat: extension bypasses sandbox through Node APIs

Mitigation:

- generated extension calls the Deno remove primitive directly
- Deno launch permissions still bind the process
- no shell execution in the extension
- no Node filesystem removal APIs in the extension

### Threat: schema drift

Mitigation:

- profile schema validates `tools.remove`
- tests assert unknown tool keys fail
- generated policy type is derived from profile types where practical

### Limit: hostile concurrent filesystem mutation

The guard is a policy and ergonomics boundary, not a complete multi-process filesystem jail. Deno permissions remain the hard containment boundary if another local process mutates paths between guard and remove.

## Tests and proof plan

Use narrow tests first, then package checks.

### Profile schema tests

Add tests for:

- `tools.remove.enabled: true` validates
- `tools.remove.recursive: false` validates
- unknown tool keys fail
- unknown remove config keys fail
- absent tools validates and resolves disabled
- generated default profile includes disabled `tools.remove` stanza for discoverability

### Template writer tests

Add tests for:

- disabled policy returns no extension args and writes nothing
- enabled policy writes `.pi/@sys/extensions/sandbox.fs.ts`
- generated extension contains no unresolved template marker
- generated extension encodes `recursive` policy
- generated extension encodes remove operation roots, not process write roots
- generated extension excludes process-only temp/cache/home roots
- template path escaping is rejected

### Resolve-run tests

Add tests for:

- enabled remove appends `--extension <path>`
- disabled remove does not append extension args
- `--no-extensions` remains present
- explicit `--extension` survives alongside `--no-extensions`
- enabled remove appends a launcher-owned prompt contract mentioning `remove`
- disabled remove does not append a remove prompt contract
- sandbox summary remains truthful and does not imply extra writes
- process write roots and remove operation roots are not accidentally collapsed

### Extension behavior tests

Prefer testing the generated extension logic through the smallest stable seam available.

Cases:

- removes a file inside cwd
- removes an empty directory with non-recursive policy
- refuses recursive input when profile recursive is false
- refuses recursive removal of final symlinks
- refuses path outside generated operation roots
- refuses raw `..` segments
- refuses glob-shaped inputs
- refuses intermediate symlink traversal
- refuses `.git` and descendants
- refuses `.pi` and descendants
- refuses wrapper runtime/log/temp descendants
- reports missing file as error

If direct extension-runtime testing is too coupled to Pi internals, extract a tiny policy guard generator/fixture test in driver-pi and keep one runtime smoke test for actual Pi loading.

## Implementation phases

### Phase 1: Type and schema surface

- Add `tools.remove` profile types.
- Add schema support.
- Add disabled profile initial/default stanza for discoverability.
- Preserve absent-means-disabled for existing profiles.
- Add schema tests.

### Phase 2: Extension template module

- Scaffold the new module boundary.
- Add FileMap bundle path.
- Add `make` and `write` helpers.
- Add generated policy injection.
- Add template writer tests.

### Phase 3: Profile run integration

- Resolve remove tool policy from profile.
- Derive remove operation roots separately from process write roots.
- Materialize extension only when enabled.
- Append launcher-owned remove prompt contract only when enabled.
- Append explicit `--extension` arg.
- Keep `--no-extensions` in place.
- Add resolve-run tests.

### Phase 4: Extension behavior proof

- Test guard logic and runtime behavior.
- Include symlink traversal and process-root separation cases.
- Run targeted tests and package check.
- Manually smoke test in a disposable file.

### Phase 5: Active profile opt-in

- Update the active profile only with human approval.
- Smallest safe YAML:

```yaml
tools:
  remove:
    enabled: true
    recursive: true
```

- Ask the human to restart Pi after profile change.

## Non-goals for v1

- No batch remove.
- No glob remove.
- No trash/recycle behavior.
- No interactive confirmation prompts.
- No generic extension loader surface.
- No broad tool policy manager.
- No replacement for `edit`/`write`.
- No relaxation of bash rules.

## Final TMIND verdict

This is stable ground only if the implementation keeps all boundaries separate:

```text
profile policy
  -> remove operation policy
  -> generated wrapper extension
  -> explicit Pi CLI extension
  -> launcher-owned prompt truth
  -> Deno process sandbox
  -> remove tool
```

The design is principled because it adds the missing primitive at the correct abstraction layer instead of punching a hole through `bash`.

S-tier acceptance requires no residue:

- no transitional names like `remove_path`
- no ambient extension discovery
- no stale extension paths in `.pi/agent/settings.json`
- no loose schema keys
- no prompt/tool mismatch
- no collapse of process write roots into remove operation roots
- no lexical-only containment without symlink policy
- no duplicated permission logic that can drift from the existing read/write resolver
