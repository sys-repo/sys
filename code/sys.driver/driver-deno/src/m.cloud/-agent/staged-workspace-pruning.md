# Staged Workspace Package-Closure Pruning

## Summary

Reduce staged Deno deploy workspaces from:

- full workspace copy

to:

- root essentials
- target package
- transitive workspace package dependencies of that target

This is package-level pruning only.

Do not:

- prune files within retained packages
- change the staged environment away from a truthful smaller workspace root
- change target build timing
- change stage entry generation

## Current State

The current stage path:

1. resolves the source workspace + target package
2. builds the target package
3. copies the whole workspace into the stage root
4. patches staged imports and generated entry files

Current whole-workspace materialization lives in:

- `src/m.cloud/m.DenoDeploy/m.stage/u.executeStage.ts`
- `src/m.cloud/m.DenoDeploy/m.stage/u.materializeWorkspace.ts`

The persisted package graph already exists at:

- `.tmp/workspace.graph.json`

It is generated by:

- `-scripts/task.prep.ts`
- `@sys/workspace` `Workspace.Prep.run()`

## Design Decisions

### Ownership

This belongs in:

- `src/m.cloud/m.DenoDeploy/m.stage/`

It does **not** belong in `m.pipeline`.
Pruning changes stage materialization, so it is a staging concern.

### Graph Authority

Use the persisted prep snapshot as the authority.

Read it from the source workspace root through the existing workspace prep graph
surface.

If the snapshot is missing:

- fail clearly
- do not silently recollect graph truth during stage

The staging contract should remain:

- prep produces the workspace graph snapshot
- stage consumes it

Snapshot freshness is therefore a prep invariant, not a staging concern.

### Pruning Unit

The retention unit is the package directory.

For target package `P`, retain:

- `P`
- every transitive workspace package dependency of `P`

Using current persisted edge direction:

- `from` = dependency
- `to` = dependent

So closure walking must:

- start from `P`
- repeatedly find edges where `edge.to === current`
- add `edge.from`
- continue until fixed point

### Path Normalization Invariant

Normalize package paths before all closure and workspace-declaration comparisons.

There are three distinct path forms entering this flow:

1. persisted graph edges use bare relative package paths
2. source/staged `deno.json.workspace` declarations may be bare or `./`-prefixed
3. the target package path resolved by `u.resolveStageTarget.ts` is dot-relative

Examples:

- `code/apps/foo`
- `./code/apps/foo`

The closure helper and staged workspace rewrite helper must compare canonicalized
forms so that:

- `code/apps/foo`
- `./code/apps/foo`

resolve to the same retained package identity.

Canonical comparison form is:

- bare relative path
- no leading `./`
- no trailing `/`

This matches the persisted graph convention.

The closure helper must normalize its target input on entry.
Do not rely on the caller to pre-normalize the target path.

When rewriting staged `deno.json.workspace`:

- compare using canonical normalized form
- preserve the original source declaration string form for each retained workspace entry

### Staged Root Truthfulness

After selective materialization, rewrite staged root `deno.json` so:

- `workspace` contains only the retained package dirs
- the retained items preserve the original order from the source workspace declaration

Do not leave pruned workspace children declared in the staged root.

### Root Essentials

Use an explicit root-file allowlist.

Do not broad-copy the workspace root.

Copy:

- `deno.json`
- `deno.lock` if present
- the `importMap` target file when referenced by staged root `deno.json`
- any root-level file that the current stage/prepare flow reads or patches directly

Do not semantically prune `imports.json` / import-map contents in this move.
Workspace reduction is controlled by package retention and staged
`deno.json.workspace`, not by import-map trimming.

## Implementation Shape

Keep the implementation split by concern.

### `u.executeStage.ts`

Orchestrates:

1. resolve target
2. read persisted graph snapshot
3. compute retained package closure
4. build target
5. materialize the reduced stage root
6. rewrite staged root workspace declaration
7. patch stage driver imports
8. write generated stage entry files

Reading the snapshot before build is preferred so missing-snapshot failures occur
before the target build work starts.

### New pure helper

Add one pure helper in `m.stage/` for closure calculation, for example:

- `u.closureFromGraph.ts`

Input:

- persisted graph
- target relative package dir

Output:

- retained package path set

This helper must stay stage-agnostic.
It should only operate on graph data and package paths.

### `u.materializeWorkspace.ts`

Keep this focused on copying/materialization only.

It should:

- copy explicit root essentials
- copy retained package dirs

It should not own closure walking logic.

### New staged-root rewrite helper

Add one helper for rewriting the staged root workspace declaration, for example:

- `u.rewriteStageWorkspace.ts`

It should:

- read staged `deno.json`
- replace `workspace` with the retained package dirs
- preserve retained declaration order
- compare normalized package paths, not raw strings

## Test Plan

### Pure closure test

Add a focused unit test for the closure helper proving:

- target with no deps retains only itself
- target with transitive deps retains the full upstream closure
- unrelated packages are excluded
- duplicate/diamond deps do not duplicate output

### Stage integration test

Extend the existing stage fixture to include:

- target app
- one retained dependency package
- one unrelated package
- a valid `.tmp/workspace.graph.json` snapshot doc that expresses the target → dependency closure

The fixture snapshot must be structurally valid.
Do not hand-write a raw `PersistedGraph` blob and treat it as a snapshot.
Use the owned snapshot helper surface so the fixture exercises the real read path.

Then assert:

- the fixture snapshot loads successfully before pruning assertions begin
- target package exists in the staged root
- retained dependency package exists in the staged root
- unrelated package does not exist in the staged root
- staged `deno.json.workspace` equals exactly the retained package declaration array
- staged workspace reloads truthfully through the existing workspace loader
- `deno.lock` is present in the staged root when present in the source workspace
- graph paths and workspace declaration paths normalize to the same retained identities

### Missing snapshot test

Add one failure test proving:

- missing `.tmp/workspace.graph.json` causes a clear prep-first failure

## Risks

Main risk is staged-root truth drift, not graph math.

Watch for:

- staged `deno.json.workspace` not matching copied dirs
- bare-relative vs dot-relative path mismatches at:
  - graph edges
  - workspace declarations
  - resolved target path
- omitted root files that the current deploy path implicitly relies on
- target `dist/` output not surviving stage materialization
- future readers misreading edge direction

## Desired Outcome

The staged deploy root remains a real workspace-shaped environment, but only the
truthful package dependency closure of the selected target is shipped.
