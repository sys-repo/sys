# TMIND: measurement and proof design

## Question
What proof shape would tell the truth about dev startup performance, and what proof shapes would mislead us?

## Required measurement split
We need at least:
1. process spawn -> ready signal
2. ready signal -> HTML 200
3. HTML 200 -> entry 200
4. entry 200 -> immediate imported modules 200
5. optional first-render / first-useful-app signal

And we need these for:
- cold run
- warm rerun

## Required counters
- count + elapsed of subprocess `deno info --json`
- count + elapsed of transport loads
- count + elapsed of CLI esbuild transforms
- npm prewarm count + elapsed
- config assembly elapsed

## Truthful proof worlds
### Must include
- at least one real outside-in app world
- at least one narrower driver-controlled world
- cold and warm separation

### Must avoid
- trivial fixture-only victory claims
- one-off stopwatch anecdote without phase timing
- measuring only server-ready time while first-load still hurts
- broad noisy aggregates that hide the bottleneck

## Output requirement
The output should let us answer:
- where the time went,
- who owned it,
- what changed after each intervention,
- whether cold or warm improved,
- whether the gain was real at the outside-in boundary.

## STIER conclusion
This lane absolutely requires instrumentation-quality proof.
Without it, the team will overfit to whichever timing anecdote confirms the current theory.
