# TMIND: transport economics

## Question
How expensive is the current transport path likely to be, and why?

## Current economic model
The transport layer is paying for at least four things:
1. resolve non-native identities
2. normalize/rewrite them into browser-servable form
3. transform TS/TSX into browser JS
4. repeat that work across many modules on first graph expansion

## Strong hotspot A — subprocess resolution
### Code anchor
- `src/m.vite.transport/u.resolve.ts`

Current resolution relies on subprocess `deno info --json`.
That is expensive relative to in-process resolution and compounds with graph width.

## Strong hotspot B — per-module CLI transform
### Code anchors
- `src/m.vite/u.wrangle.ts`
- `src/m.vite.transport/u.load.ts`

Current transform path prefers CLI esbuild when `ESBUILD_BINARY_PATH` is present.
That means each transported non-JS module can pay process-spawn overhead.

## Why graph width matters
A small per-module tax becomes a large first-load penalty when the early graph includes:
- many `@sys/std/*` leaves
- UI packages
- devharness surfaces
- transitive browser-facing TS modules

## Why this seam is so dangerous
It can produce a misleading profile:
- server “starts” acceptably,
- but first meaningful page load still feels bad,
- because the expensive work is deferred into the first graph walk.

## What would falsify this concern?
Only measurement showing:
- transport counts are small,
- CLI transform time is negligible,
- or first-load pain is dominated elsewhere.

Absent that, transport remains the highest-confidence driver-owned bottleneck.

## STIER conclusion
Transport economics are currently the most suspicious part of the whole system.
If the first packet's measurements confirm this, deeper optimization should focus here before broader tuning theater.
