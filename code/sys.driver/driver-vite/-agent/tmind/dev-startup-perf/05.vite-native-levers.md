# TMIND: Vite-native levers

## Question
What upstream-supported Vite levers are we currently underusing before reaching for deeper custom work?

## Candidate lever A — `dev.warmup`
Vite 8 supports explicit warmup and defaults to no warmup.
That means the current line is likely leaving some startup/perceived-first-load improvement on the table.

### Likely use
- warm HTML entry
- warm main app entry
- perhaps warm only the minimal/default path first

## Candidate lever B — `optimizeDeps.include`
Vite itself recommends explicit include lists to improve cold start.
The obvious current candidates are:
- `react`
- `react-dom`
- `react-dom/client`

Potentially later:
- tiny additional hot npm surfaces justified by measurement

## Candidate lever C — dep optimizer posture discipline
We should ensure we are not accidentally withholding optimization from obvious stable npm deps while pushing all cold cost into runtime transport.

## What these levers can and cannot do
### They can
- improve cold start for stable npm deps
- improve perceived responsiveness
- reduce some repeated optimizer guesswork

### They cannot
- remove subprocess transport cost for transported TS/TSX modules
- fix broad app graph weight
- solve Deno subprocess resolution by themselves

## Hard STIER rule
Use Vite-native levers first where they honestly apply, but do not overclaim them.
If transport dominates, these are supportive optimizations, not the core answer.

## STIER conclusion
Vite-native levers are a good first packet because they are:
- low risk,
- upstream-aligned,
- easy to measure,
- and capable of real modest wins.
But they should not be mistaken for a substitute for deeper transport work if the transport data comes back heavy.
