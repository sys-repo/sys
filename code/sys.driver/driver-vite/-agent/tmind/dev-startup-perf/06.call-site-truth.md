# TMIND: call-site truth

## Question
How much of the perceived slowness is being truthfully caused by the external app rather than the driver?

## Current call-site signal
In `agent-projects/code/projects/slc-data`, the supposedly lightweight splash path still imports:
- `@sys/ui-react-devharness`

That appears to be for `useKeyboard()` in `src/-test/entry.splash.tsx`.

## Why this matters
That import is unlikely to be free.
It can widen the startup graph with:
- additional UI/runtime modules
- dev-oriented surfaces
- transitive `@sys/*` browser modules

## Hard truth
If the default splash path is not actually minimal, then some of the perceived slowness belongs to the app.
That does not excuse driver cost, but it does constrain what a driver-only optimization can achieve.

## Anti-falsehood rules
- Do not use a pathologically heavy app path to declare the driver slow in principle.
- Do not use a trivial fixture to declare the driver fast in practice.
- Keep one real outside-in call-site truth world in the loop.

## Correct interpretation
The app likely compounds the driver's cold-path costs.
That means the honest optimization stance is two-track:
1. improve driver startup/transport cost,
2. later review the call site's splash/default graph weight.

## STIER conclusion
Call-site truth is necessary as a correction factor.
Without it, we risk either unfairly blaming the driver or falsely congratulating it with toy proofs.
The current `slc-data` startup path does not look like a clean minimal baseline.
