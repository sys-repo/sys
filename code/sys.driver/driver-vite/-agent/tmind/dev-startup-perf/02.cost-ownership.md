# TMIND: cost ownership

## Question
Who owns which classes of dev-startup cost?

## Driver-owned costs
- startup authority projection/delivery
- config assembly performed by `Vite.Config.app(...)`
- workspace alias derivation
- optimize-import rule derivation
- npm prewarm policy
- transport resolution orchestration
- transport transform strategy

## Vite-owned / upstream costs
- core dev server boot
- dep optimizer internals
- module graph invalidation behavior
- OXC/Rolldown transform behavior
- warmup semantics

## Deno-owned costs
- `deno info --json` resolution behavior and latency
- npm/JSR materialization behavior
- cache population behavior
- subprocess launch overhead for Deno commands

## App-owned costs
- entry graph breadth
- heavy early imports in splash/default path
- dev-only tooling imported in normal startup path
- chunk/dependency posture at the call site

## Ecosystem/package-surface-owned costs
- hot `@sys/*` browser packages publishing TS-heavy rather than browser-ready surfaces
- broad package runtime graphs that increase transport work

## Hard ownership rules
- Do not blame Vite for driver subprocess choices.
- Do not blame the driver for app splash-path overreach.
- Do not blame the app for resolution work introduced by driver policy.
- Do not claim a driver win if the gain only came from weakening published-boundary truth.

## Ownership risk
The most dangerous failure mode is owner confusion leading to fake optimizations:
- tuning call-site code when transport dominates,
- rewriting transport when the app is importing devharness in the default splash path,
- or over-customizing the driver when Vite-native levers would have been enough.

## STIER conclusion
This perf problem is multi-owner.
But the most actionable near-term owner is still `driver-vite`, because it directly controls:
- npm prewarm breadth,
- config-time work,
- transport orchestration,
- and whether each transformed module pays a subprocess tax.
