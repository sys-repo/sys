# STIER cross-review: `driver-vite` dev startup perf TMIND set

## Purpose
Stress the seven perspectives against each other, expose duplication, test for missing dimensions, and collapse weak assumptions.

## Cross-findings

### 1. The set is well-shaped; no major missing dimension
The seven dimensions cover:
- what happens
- who owns it
- what repeats
- what is expensive
- what upstream knobs exist
- what the app contributes
- how truth will be measured

A separate “prioritization” perspective is not required yet because ranked action can be distilled from the current set.

### 2. Transport dominates suspicion, but not yet proof
Across:
- mechanism breakdown
- cache posture
- transport economics
- measurement/proof design

one theme dominates: transport is the strongest current bottleneck hypothesis.

Hard STIER correction:
- this is a high-confidence suspicion,
- not yet a measured fact.

So the next packet must instrument before redesign.

### 3. Warm-cache comfort is the main false-friend risk
Across:
- cache posture
- call-site truth
- measurement/proof design

a likely failure mode is declaring victory from warm reruns while the user's actual stop/start cold experience remains bad.

### 4. There are two real worlds, not one
The system truth is a conjunction of:
1. driver-owned cold-path cost
2. app-owned startup graph breadth

Any review that collapses those into one source of pain will produce fake action.

### 5. Vite-native levers are justified, but must stay modest in claim
Across:
- Vite-native levers
- transport economics

`dev.warmup` and `optimizeDeps.include` are good first interventions.
But if the measurements show heavy transport subprocess cost, they are helpers, not the center of gravity.

## Weak assumptions exposed
### Assumption A
"The `Try` runtime bug may have been the main reason startup felt bad."

Correction:
- it may have obscured settling/warm behavior,
- but the code still shows plausible expensive cold-path work.

### Assumption B
"If server ready is fast, startup is fine."

Correction:
- first-load transport can still dominate the user experience.

### Assumption C
"Broad npm prewarm is definitely the main problem."

Correction:
- broadness is real,
- magnitude is still unproven.

## STIER verdict
The dimension set survives hard review.
The strongest disciplined conclusion is:
- do not redesign yet,
- instrument first,
- try Vite-native levers next,
- keep transport redesign as the likely deeper lane if measurements confirm it.
