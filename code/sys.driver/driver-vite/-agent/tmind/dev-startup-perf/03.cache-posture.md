# TMIND: cache posture

## Question
What work is cached today, at what layer, and what still repeats on cold restart?

## Present cache layers
### Vite/dev server cache
- standard Vite cache behavior under `node_modules/.vite`
- helps after initial optimizer/materialization work
- does not automatically erase driver transport cost for arbitrary JSR/remote TS modules

### Driver startup delivery identity
- startup delivery path is deterministic
- avoids residue drift and improves stability
- good for correctness, but not itself a large performance cache

### Transport in-memory cache
- current transport holds in-memory resolution state within a running server process
- useful for repeated requests during one session
- lost across restart

### Deno/npm/JSR materialization cache
- can make later resolution/materialization cheaper
- but may still leave driver orchestration and transform work active on restart

## Likely uncached / weakly cached work
- subprocess `deno info --json` cost across fresh dev restarts
- per-module CLI transform cost across fresh dev restarts
- broad npm prewarm decision work at startup
- first-hit transport transform for browser-served TS/TSX modules

## Cold vs warm truth
A warm rerun may feel much better while the architecture is still too expensive on cold start.
That means:
- warm green is not enough,
- cold/warm split must be measured separately,
- and in-process cache benefits must not be mistaken for real restart wins.

## Cache posture assessment
The current cache posture is respectable for correctness and session-local reuse, but weak for the exact user complaint:
- stop/start dev repeatedly,
- hit the app,
- wait through first-load transport work again.

## STIER conclusion
Cache posture likely explains part of the gap between “architecture is cleaner” and “dev still feels slow.”
The current line appears stronger on determinism than on cross-restart reuse.
That is acceptable for correctness closeout, but not yet a compelling final perf posture.
