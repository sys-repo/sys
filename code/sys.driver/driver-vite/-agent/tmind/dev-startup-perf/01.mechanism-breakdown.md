# TMIND: mechanism breakdown

## Question
What exact phases make up current cold dev startup and first useful page load?

## Cold path
1. `deno run @sys/driver-vite/main`
2. entry/arg handling in `src/-entry/*`
3. wrangle phase in `src/m.vite/u.wrangle.ts`
   - locate config/package context
   - derive runtime paths
   - resolve `vite`
   - resolve esbuild binary
   - build startup env/import-map handoff
4. startup authority projection in `src/m.vite.startup/u.projection.ts`
5. startup delivery write in `src/m.vite.startup/u.delivery.ts`
6. Vite config load
7. app config assembly in `src/m.vite.config/u.app.ts`
   - workspace alias discovery
   - optimize-import rule derivation
   - plugin assembly
8. plugin startup hooks
   - specifier rewrite prep
   - npm prewarm
9. Vite server ready
10. browser requests HTML
11. browser requests app entry
12. transport resolves non-native identities in `src/m.vite.transport/*`
13. transported TS/TSX modules transformed for browser execution
14. module graph evaluates
15. UI renders

## Mechanism implications
- There are at least three user-visible waits:
  - process-to-ready
  - ready-to-first-entry-200
  - first-entry-200 to first useful render
- A “Vite is ready” log does not mean the expensive work is done.
- The first browser hit is still carrying major transport and transform cost.

## Mechanism suspicion ranking
1. transport resolution/transform after first entry request
2. config/plugin startup work before ready
3. startup projection/delivery overhead

## STIER conclusion
Mechanically, the current line is most vulnerable after server readiness, not only before it.
If the user experience is “ages,” the likely truth is:
- some cost exists pre-ready,
- but the dominant pain may still be entry-triggered transport/transform work on first load.
