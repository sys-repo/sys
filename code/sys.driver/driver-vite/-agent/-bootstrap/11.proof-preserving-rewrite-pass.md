## Distilled question
Which parts can be safely simplified without reopening the frontier?

## Distilled answer
The proven line should **not** be rewritten as one big act of courage.
It should be rewritten by separating:

- what is already earned and should stay fixed
- what is obviously too broad but can be simplified without changing behavior
- what is architecturally wrong but too risky to change without focused proof
- what must be isolated before touching because it sits on a live compatibility boundary

The single most important rewrite posture is:

**preserve external proof first, then compress internal shape.**

Not:
- make it cleaner and hope the proofs still mean the same thing

The real outcome of this pass is a ranked rewrite map.

---

## Rewrite doctrine for this frontier
The earlier passes now imply a very strict doctrine:

1. do **not** reopen the “what works” frontier
2. do **not** treat rescue residue as sacred
3. do **not** delete risky seams before replacement proof exists
4. do extract the smaller true contract and move toward it in proof-preserving stages

So this pass is not asking:
- what is ugly?

It is asking:
- what can be changed while keeping the proven Vite 8 line intact?

That is a different discipline.

---

## Ranking frame
This pass uses four rewrite buckets:

### 1. Keep as-is for now
Touches here are more likely to reopen the frontier than to buy real architectural gain.

### 2. Simplify safely
These can usually be improved while preserving behavior because the essential shape is already known and the change is mostly factoring, naming, or internal separation.

### 3. Redesign with tests first
These are real architectural liabilities, but they sit too close to the proof line to change casually.
They need focused tests or probes before replacement.

### 4. Isolate before touching
These are seams whose current entanglement is itself the risk.
Before changing their behavior, isolate them behind sharper boundaries.

**Why isolate first:**

Do not mix:
- bootstrap architecture changes
- transport changes

**Classification:**
- isolate before touching

## The first safe rewrite sequence
This pass implies a practical rewrite order.

## Step 1
Freeze the earned invariants:
- `--configLoader=native`
- bootstrap remains its own seam
- wrangle remains the command seam
- transport stays out of scope

## Step 2
Refactor `u.bootstrap.ts` into explicit internal stages while preserving runtime behavior:
- projection
- delivery
- cleanup beneath delivery

## Step 3
Only after that isolation, add focused tests/probes for:
- deterministic artifact identity
- narrower emitted authority payload
- ranked root/nearest policy
- reduced dependency projection

## Step 4
Then redesign the risky pieces one by one.

That is the proof-preserving route.

---

## The main anti-pattern to reject
The main anti-pattern for this pass is:

**cosmetic architectural rewrite ahead of proof isolation.**

That is how teams create brand-new legacy while telling themselves they are simplifying.

In this frontier, the correct posture is:
- keep the earned launch lane
- split concepts first
- replace liabilities only after focused proof exists for each one

---

## Distilled engineering conclusion
The rewrite should begin by protecting the earned boundaries and reducing internal entanglement.
Not by changing the deepest runtime assumptions on day one.

So the clean summary is:

**first separate the proven seam from its residue, then replace the residue under proof.**

That is the proof-preserving doctrine for this refactor.
