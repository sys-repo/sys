## Distilled question
What in this seam has stable identity, and what is random or run-specific?

## Distilled answer
The current bootstrap seam has a **stable conceptual contract** but an **unstable operational identity**.

That is the central determinism problem.

In other words:
- the seam knows *roughly what authority it wants to project*
- but it does not give that authority a stable runtime identity across equivalent runs

The main determinism break is not subtle:

**equivalent inputs currently produce new bootstrap artifact identities every run.**

That is anti-cache, anti-reuse, and anti-final-shape.

---

## Determinism standard used in this pass
For this note, a seam is deterministic when:

- equivalent inputs produce the same effective identity
- runtime handles are stable enough to support cache reuse and reasoning
- representation does not churn merely because the process was started again

A seam is identity-unstable when:
- paths, file URLs, or artifact names change per run without a semantic reason
- consumers/tooling cannot cheaply recognize equivalence
- the runtime boundary is defined by transient artifacts rather than stable projection

---

## The identity surfaces in play
To understand this seam, we need to distinguish between several kinds of identity.

### 1. Semantic identity
What bootstrap *means* to provide.

### 2. Content identity
What the projected authority payload actually contains.

### 3. Artifact identity
What files/paths/URLs are used to deliver that authority.

### 4. Command identity
What concrete child command line is launched.

The deepest current issue is:
- semantic identity is relatively stable
- content identity is partly stabilizable
- artifact identity is unstable
- command identity therefore also churns

## Stable identity surfaces already present
These are the parts of the seam that are already deterministic in principle or close to it.

## 1. Version-gated activation rule
**Observed shape:**
- activate only when `viteMajor(vite) >= 8`

**Determinism status:**
- stable

**Why:**
For a given Vite specifier, the activation rule is deterministic.
There is no run-time randomness here.

---

## 2. Authority discovery algorithm
**Observed shape:**
- nearest `deno.json`
- nearest workspace-root `deno.json`
- nearest `package.json`

**Determinism status:**
- stable for a fixed filesystem state

**Why:**
Given the same directory tree and files, the walk-up result is deterministic.

**Caution:**
This says nothing about whether the discovered authority model is *correctly ranked*.
It only says the lookup process itself is repeatable.

---

## 3. Import extraction and normalization
**Observed shape:**
- string-record normalization
- deterministic filtering of import-map entries

**Determinism status:**
- stable

**Why:**
For the same underlying config documents, the normalized import records are deterministic.

---

## 4. Import ordering
**Observed shape:**
- `sortImports(...)`

**Determinism status:**
- stable and intentionally stabilizing

**Why:**
This is one of the clearest signals that the seam already partly knows determinism matters.
It canonicalizes output ordering explicitly.

## 5. Vite specifier derivation in `u.wrangle.ts`

- resolve Vite version from package authority
- emit `npm:vite@<version>`

**Determinism status:**
- stable for a fixed dependency state

**Why:**

## Identity-unstable surfaces
These are the major determinism breaks.

## 1. Random bootstrap import-map artifact path
**Observed shape:**
- `.vite.bootstrap.${crypto.randomUUID()}.imports.json`

**Determinism status:**
- unstable by construction

**Why:**
Even if the authority content is identical across runs, the artifact path changes every time.
That means:
- new file path
- new `--import-map=...` arg
- new command identity
- likely new file-based cache key surface

**Engineering implication:**
This is the single clearest operational identity failure in the current seam.

---

## 2. Random bootstrap shim module path
**Observed shape:**
- `.vite.bootstrap.${crypto.randomUUID()}.module-sync-enabled.mjs`
- used via file URL in imports

**Determinism status:**
- unstable by construction

**Why:**
This creates a fresh file URL every run.
For any system sensitive to module URL identity, that is effectively a new thing every time.

**Engineering implication:**
This is likely even worse than the import-map path churn, because it changes a specifier value inside the projected authority itself.

---

## 3. Child command identity churn
**Observed shape:**
- `Wrangle.command(...)` includes `--import-map=<generated path>`

**Determinism status:**
- unstable because bootstrap artifact path is unstable

**Why:**
Even if every meaningful logical input is unchanged, the launched command string changes because the import-map path changes.

**Engineering implication:**
This weakens any chance of stable reuse keyed off exact command or startup input identity.

---

## 4. Cleanup-based ephemeral delivery
**Observed shape:**
- create temp artifacts, then remove them after use

**Determinism status:**
- unstable lifecycle posture

**Why:**
This prevents the seam from even pretending to offer stable persistent identity across equivalent runs.
The artifacts are intentionally disposable and replaced with new ones next time.

**Engineering implication:**
This makes the bootstrap delivery mechanism inherently cold-start oriented rather than reuse oriented.

---

## 5. Mixed identity classes collapsing into one output artifact
**Observed shape:**
- root + nearest + runtime + synthetic shim all collapsed into one generated import map

**Determinism status:**
- partly stable in content, unstable in delivery identity

**Why:**
Even where content can be made canonical, the current delivery model forces the final operational identity to churn.

**Engineering implication:**
Deterministic content alone is not enough if artifact identity remains unstable.

---

## The central determinism split
The current seam should be understood through this split:

### Stable enough now
- discovery logic
- normalization logic
- ordering logic
- version gate logic
- much of the underlying projected content, in principle

### Unstable now
- file paths
- file URLs
- import-map handle identity
- command-line identity
- delivery lifecycle

This matters because the seam's **observable runtime identity** is what downstream tools actually see.
And that is currently unstable.

---

## Equivalent-run thought experiment
This is the decisive test.

Suppose all of the following remain unchanged:
- cwd
- filesystem state
- `deno.json`
- import-map content
- Vite version
- package.json dependencies
- bootstrap dependency lists

Then ask:

### Should bootstrap be the same thing?
Conceptually: yes.

### Is bootstrap the same thing now?
Operationally: no.

Because the seam currently changes:
- import-map filename
- shim filename
- shim file URL
- command-line import-map arg

So equivalent semantic inputs do **not** yield equivalent operational identity.
That is the core failure of determinism.

## Why this matters beyond aesthetics
This is not just about “ugly temp filenames.”
The determinism failure has direct architectural consequences.

## 1. Cache posture
If equivalent runs look new at the artifact/command level, caches are harder to hit or reason about.

## 2. Debuggability
A changing bootstrap identity makes it harder to compare runs and reason about whether behavior changed because:
- the world changed
or
- the seam just regenerated itself differently

## 3. Consumer legitimacy
A stable driver seam feels like infrastructure.
A random temp-file seam feels like a workaround.

## 4. Refactorability
A deterministic model is easier to factor cleanly because you can name stable boundaries and stable contracts.

---

## Contrast with cleaner identity models

## `m.vite.config.workspace`
This module has a much cleaner identity posture.
Its output is derived deterministically from:
- workspace file state
- package exports

It does not manufacture run-specific artifact identities.
That makes it much closer to a final-shape seam.

## `@deno/vite-plugin`
Within its own boundary, it also has a cleaner identity posture:
- config discovery
- loader creation
- caches keyed per environment

Even if it is not solving your seam, it is not centered on random artifact identity.
That contrast is useful.

## What the essential contract implies about identity
From pass 04, the essential contract is:
- provide the minimum child-consumable bootstrap authority

That contract says nothing about random UUIDs.
It says nothing about temp artifact churn.
It says nothing about cleanup-first delivery.

So determinism is not blocked by the contract.
It is blocked by the current implementation strategy.

That is important.
The current anti-determinism is not essential.
It is contingent.

---

## Future-shape determinism requirements
Without jumping fully into redesign, this pass implies the replacement seam must satisfy:

### 1. Equivalent inputs → equivalent bootstrap identity
Not just equivalent content.
Equivalent operational identity.

### 2. Delivery identity should be stable or derivable
If artifacts remain, they should be:
- stable-path
or
- content-addressed
or
- otherwise deterministically named

### 3. Command identity should stop churning for equivalent runs
If the bootstrap handle is stable, the child command identity becomes more stable too.

### 4. Synthetic identity must be minimized
The less the seam depends on fabricated runtime artifacts, the less identity churn it can create.

---

## Distilled engineering conclusion
The determinism problem in the current line is not that the seam lacks order.
It already has some order.

The problem is that:

**the seam stabilizes content while destabilizing runtime identity.**

That is the wrong half to leave unstable.

The deepest takeaway from this pass is therefore:

- the next redesign must treat stable artifact/handle identity as a first-class requirement
- otherwise the seam will remain rescue-shaped even if the code gets smaller

- the current bootstrap seam is not fundamentally non-deterministic in logic, but it is non-deterministic in delivery identity
- that is enough to make it feel and behave like unstable infrastructure
