## Distilled question
What is the minimum real contract this seam must satisfy?

## Distilled answer
The essential contract of `u.bootstrap.ts` is:

**for Vite 8+ child CLI execution under Deno, provide just enough bootstrap-time import authority for the child process to evaluate and enter the Vite config/runtime path truthfully, then get out of the way.**

That is the contract.
Not:
- recreate the whole consumer environment
- preserve arbitrary config document shape
- encode every workaround discovered during the probe line
- own app/plugin resolution generally

The seam exists only to bridge the bootstrap gap between:
- the consumer's declared authority
and
- the Vite 8 child runtime's startup needs under Deno

---

## First-principles contract statement

### One-sentence form
`u.bootstrap.ts` must project the minimum Deno-aware authority required for a Vite 8 child process to start and evaluate its config/runtime path under Deno.

### Slightly expanded form
When `Wrangle.command(...)` launches a Vite 8 child via `deno run`, the child must receive a bootstrap authority environment that is sufficient for:
- Vite config loading
- bootstrap-time runtime imports
- required toolchain subpaths/builtins

without forcing the child to guess or ignore the consumer's Deno authority.

Once that is achieved, bootstrap has completed its job.

---

## Where the contract begins and ends

## Begins
The contract begins at:
- `Wrangle.command(...)`
- where the child CLI command is assembled
- where an optional bootstrap import-map path is passed via `--import-map=...`

This is important because `u.bootstrap.ts` is not an abstract helper.
It exists because `Wrangle.command(...)` must produce a runnable child command line for Vite 8 under Deno.

## Ends
The contract ends when:
- the Vite child has enough authority to enter its config/bootstrap/runtime path
- `build(...)` or `dev(...)` can proceed normally
- app/plugin/runtime resolution can be handled by the proper downstream seams

This means bootstrap is **not** responsible for:
- broad app import rewriting
- long-lived consumer module resolution policy
- general workspace alias behavior once the proper runtime seams are active

Bootstrap is a startup-enablement seam only.

---

## Essential inputs
These are the inputs the contract actually needs, at minimum.

### 1. Consumer execution root
**Why needed:**
Bootstrap needs a starting point for locating the relevant consumer/workspace authority.

### 2. Vite version/specifier
**Why needed:**
The seam is explicitly Vite-8+-scoped.
Without version knowledge, the bootstrap cannot know whether it should activate.

### 3. Consumer-declared Deno authority
At minimum this means whatever declared import authority is genuinely relevant to child startup.

**Why needed:**
The point of the seam is to preserve Deno-owned startup truth rather than let the child run in a detached Node/Vite-only authority world.

### 4. Bootstrap-time Vite/runtime requirements
This includes only the specific specifiers the child needs in order to begin truthful evaluation.

**Why needed:**
The child needs more than the consumer's own app imports; it also needs its own runtime/toolchain imports to work under Deno.

---

## Essential outputs
These are the outputs the contract actually needs, at minimum.

### 1. A bootstrap authority handle consumable by `deno run`
In the current line, this is an import-map path.
At contract level, what matters is only this:

- the child command can consume it directly
- it supplies the minimum startup authority required

Whether this remains a generated file, a deterministic file, or some other representation is an implementation question, not a contract truth.

### 2. A cleanup/disposal path if the implementation uses generated state
This is only essential **if** generated state remains part of the design.
It is not essential at the abstract contract level.

That distinction matters.

---

## Essential invariants
These are the non-negotiable invariants implied by the proven line.

### 1. Vite 8+ bootstrap authority is its own seam
Bootstrap authority must remain distinct from app/plugin resolution concerns unless a replacement seam is equally proven.

### 2. The seam must be Deno-truthful
The child must not bootstrap in a way that discards or misrepresents the consumer's Deno authority.

### 3. The seam must be minimal
Bootstrap must supply only startup-essential authority, not carry broad configuration residue by default.

### 4. The seam must be deterministic in principle
Even if the current implementation is not yet deterministic, the contract itself demands a stable identity model for equivalent runs.
A non-deterministic bootstrap is not a clean final contract.

### 5. The seam must preserve the proven Vite 8 line
Any replacement must still support the earned Vite 8 config/bootstrap/runtime path under Deno.

### 6. The seam must not pretend to solve all resolution concerns
Its job is startup enablement, not total system ownership of module resolution.

---

## What is **not** part of the essential contract
This is the most important distillation step.

The following are **current implementation choices**, not contract truths:

### Not essential
- random UUID filenames
- temp artifact placement in package roots
- whole-document JSON carry-through
- broad root + nearest JSON merge
- cleanup callback as a first-class abstraction
- package.json probing in its current form
- the exact current hard-coded dependency list shapes
- preserving the generated import-map mechanism itself

These may have been useful in the probe line.
They are not what the seam fundamentally is.

---

## Contract restated as responsibilities
The essential responsibilities of `u.bootstrap.ts` can be stated precisely as:

### Responsibility 1
Detect whether the current Vite child launch crosses into the Vite-8 bootstrap seam that needs special handling.

### Responsibility 2
Project the minimum startup authority the child needs from consumer/workspace truth plus any unavoidable Vite-runtime truth.

### Responsibility 3
Hand that authority to the child command in a form the child runtime can actually consume.

### Responsibility 4
Stop there.
Do not absorb broader app/plugin/runtime concerns that belong elsewhere.

That is the contract in operational terms.

## Relationship to adjacent modules

### `u.wrangle.ts`
This file clarifies the contract sharply.
`u.bootstrap.ts` is only used so that `Wrangle.command(...)` can produce a valid child `deno run ... vite ...` invocation.
That means bootstrap is a **command-preparation seam**, not a general runtime subsystem.

### `u.build.ts` and `u.dev.ts`
These files also clarify the contract.
They do not care about bootstrap internals.
They only need:
- command args/env
- a disposal path
- a child runtime that actually starts

That strongly implies the abstract bootstrap contract is small.

### `m.vite.config.workspace`
This is useful as a model because it shows what a cleaner authority-projection seam looks like:
- declared truth in
- deterministic projection out

That is closer to the correct contract style than the current rescue-heavy bootstrap shape.

## Distilled engineering conclusion
The most important result of this pass is:

**the essential contract is much smaller than the current implementation.**

The contract does **not** require:
- rescue-shaped temp artifacts
- broad document carry-over
- blocker-history encoded in runtime form

It only requires:
- a truthful, minimal, child-consumable bootstrap authority projection for Vite 8 startup under Deno

That should become the anchor for every later pass.
