## Distilled question
What kinds of authority are being mixed here, and should they be?

## Distilled answer
`u.bootstrap.ts` is currently mixing **five different authority classes** into one synthetic bootstrap document:

1. consumer-declared Deno authority
2. workspace-root Deno authority
3. bootstrap-owned Vite runtime authority
4. Node/package metadata authority
5. generated synthetic artifact authority

The first three can be legitimate in principle.
The last two are where the authority model becomes suspect.

The core problem is not merely that there are multiple authority sources.
The core problem is that the module currently treats them as if they can all be merged into a single bootstrap-world artifact without a sharp hierarchy.

That is not a clean long-term authority model.

---

## Why this pass matters
The current bootstrap seam is fundamentally an **authority-projection** problem.
If the authority model is wrong or blurred, then everything else becomes unstable:
- cache identity
- consumer truthfulness
- runtime predictability
- factoring into the correct Vite hook/seam

So this pass is not secondary.
It is one of the core architecture passes.

---

## Authority classes present in the current line

## 1. Consumer-declared Deno authority
**Observed sources:**
- nearest `deno.json`
- inline `imports`
- referenced import-map file
- its `imports`

**What this authority means:**
This is the consumer's declared Deno resolution truth for the current package/app boundary.

**Examples:**
- bare import mappings
- `jsr:`/`npm:` aliases declared by the consumer
- package-local import authority decisions

**Classification:**
- primary truth

**Should it be present?**
- yes
- absolutely

**Why:**
If bootstrap does not preserve this authority, the Vite child is no longer booting in the consumer's real declared world.
That would violate the essential contract.

---

## 2. Workspace-root Deno authority
**Observed sources:**
- nearest workspace root `deno.json`
- workspace-root import-map path and/or imports

**What this authority means:**
This is broader Deno authority declared at workspace level rather than package-local level.

**Classification:**
- derived truth
- but often legitimately derived truth

**Should it be present?**
- maybe, but only in a precisely justified form

**Why:**
In a real Deno workspace, package-local consumer truth may inherit or rely on root/workspace authority.
So workspace-root authority is not automatically accidental.

**Current concern:**
The issue is not that workspace authority exists.
The issue is that `u.bootstrap.ts` currently merges it broadly with nearest authority without making the hierarchy explicit enough.

**Architectural read:**
- workspace-root authority may belong in the model
- but only as an explicitly ranked/derived authority layer, not as an undifferentiated merge source

---

## 3. Bootstrap-owned Vite runtime authority
**Observed sources:**
- `vite`
- `vite/internal`
- `vite/module-runner`
- `fs -> node:fs`
- `path -> node:path`
- `zlib -> node:zlib`
- bootstrap dependency lists

**What this authority means:**
This is not consumer-declared application authority.
This is authority required by the Vite child runtime itself in order to start under Deno.

**Classification:**
- owned toolchain/runtime authority
- derived but legitimate bootstrap authority

**Should it be present?**
- yes, in some form

**Why:**
The child must have enough runtime authority to evaluate Vite 8 config/bootstrap/runtime paths.
That is part of the proven contract.

**Current concern:**
The presence of this authority is not the problem.
The current problem is that its shape is manual, broad, and blocker-history-driven.

**Architectural read:**
This authority class is real, but it should become:
- explicit
- minimal
- stable
- ranked beneath consumer truth, not blended into it casually

---

## 4. Node/package metadata authority
**Observed sources:**
- upward `package.json` search
- dependency/devDependency probing
- consumer version detection for `@vitejs/plugin-react`
- `u.wrangle.ts` package-anchor logic for vite/esbuild discovery

**What this authority means:**
This is authority inferred from Node package metadata, not from Deno import authority.

**Classification:**
- ambient authority
- interop authority
- not primary truth

**Should it be present?**
- only reluctantly and explicitly

**Why:**
Some of this may be necessary because the child runtime crosses into npm/Vite territory.
But this is not the same class of truth as the consumer's Deno authority.

**Current concern:**
The current seam lets this authority participate in shaping bootstrap behavior without a sufficiently explicit hierarchy.
That is dangerous because ambient package metadata can silently become resolution truth.

**Architectural read:**
This authority class should be treated as:
- secondary
- interop-only
- tightly constrained
- never mistaken for the primary consumer authority model

---

## 5. Generated synthetic artifact authority
**Observed sources:**
- generated `.vite.bootstrap.*.imports.json`
- generated `.vite.bootstrap.*.module-sync-enabled.mjs`
- file URL for shim module

**What this authority means:**
This is authority that did not exist in the consumer's world and is fabricated by the driver in order to make the child runtime boot.

**Classification:**
- synthetic authority
- implementation artifact authority
- non-truth-bearing authority

**Should it be present?**
- only if there is no cleaner proven alternative
- otherwise no

**Why:**
This is not declared truth.
It is generated compatibility scaffolding.
That makes it the weakest authority class in the whole model.

**Current concern:**
The current implementation lets synthetic artifact authority become the operational center of the seam.
That is the biggest authority-model smell in the file.

**Architectural read:**
Synthetic artifact authority should be assumed temporary and should never be allowed to define the conceptual model.

---

## Actual hierarchy the model should respect
The current line does not express this hierarchy sharply enough.
A correct top-down hierarchy should look closer to this:

### Level 1 — Primary authority
- consumer-declared Deno authority

### Level 2 — Explicitly derived authority
- workspace-root Deno authority, only insofar as the consumer truth actually derives from it
- bootstrap-owned Vite runtime authority, only insofar as the child startup contract requires it

### Level 3 — Interop authority
- Node/package metadata needed for explicitly justified npm/Vite interop seams

### Level 4 — Implementation authority
- generated synthetic artifacts, only if unavoidable

This hierarchy is the real missing shape.
The current file behaves more like:
- gather several authority sources
- merge enough of them into a bootstrap artifact
- hope the child world becomes bootable

That was acceptable in the probe line.
It is not the final model.

---

## Where the current model blurs authority incorrectly

## 1. Root and nearest are merged too early
The current code merges root and nearest import/json payloads before later design questions are answered.
That means hierarchy is lost too soon.

### Why this is a problem
Once merged, it becomes hard to answer:
- which authority actually justified this mapping?
- was this inherited from workspace truth or locally declared?
- is this broad carry-over or truly required startup authority?

### Correct read
The scar is not “multiple authority sources exist.”
The scar is “the hierarchy is collapsed into a synthetic merge too early.”

---

## 2. Toolchain authority is mixed into consumer authority via the same import-map artifact
The current line writes one final import-map payload that includes:
- consumer/workspace imports
- Vite runtime imports
- node builtin mappings
- injected shim module mapping

### Why this is a problem
Those are not the same kind of truth.
Putting them in one document may be operationally convenient, but conceptually it blurs:
- user-declared authority
- driver-owned startup authority

### Correct read
Even if one final consumable artifact remains, the internal model should still keep these authority classes separate and ranked.

---

## 3. Package metadata participates too loosely in bootstrap decisions
`consumerVersion(...)` allows Node metadata to decide whether more bootstrap authority is injected.

### Why this is a problem
This is ambient authority affecting bootstrap shape.
It may be justified, but it should be treated as interop-only and scrutinized heavily.

### Correct read
This is not primary truth and should never be allowed to masquerade as it.

---

## 4. Synthetic artifact authority is too central
The generated bootstrap files are currently the form in which authority becomes operational.

### Why this is a problem
This gives implementation scaffolding too much conceptual weight.
The seam starts to look like “the generated bootstrap files are the model,” when they should at most be a delivery mechanism.

### Correct read
The authority model should be defined independently of any generated file strategy.

---

## Contrast with cleaner authority models

## `m.vite.config.workspace`
This is a much cleaner authority model because it does not pretend all authority classes are equal.
It primarily uses:
- workspace package declarations
- package exports
- deterministic alias projection

That is a much clearer chain of truth:
- declared package truth in
- alias projection out

It is useful precisely because it does **not** center a synthetic merged bootstrap world.

## `@deno/vite-plugin`
This plugin also has a cleaner authority stance within its intended seam.
It finds Deno config and uses a Deno loader in Vite's plugin pipeline.
It does not claim to solve config/bootstrap.
That limitation is honest.

The lesson here is important:
- a tool can have a narrow authority model and still be correct
- breadth is not the same as correctness

---

## Authority-model restatement in contract terms
The essential contract from pass 04 can now be sharpened:

The bootstrap seam must not answer:
- “what is every authority source we can gather?”

It must answer:
- “what minimum ranked authority set is required for truthful Vite 8 child startup under Deno?”

That is the authority-model correction.

---

## What the future authority model should aim for
Without jumping fully into redesign yet, this pass implies the following target qualities:

### 1. Ranked authority, not flat merge
The model must preserve the distinction between:
- primary declared consumer truth
- derived workspace truth
- bootstrap-owned runtime truth
- interop-only metadata
- synthetic delivery artifacts

### 2. Projection over preservation
The model should project the authority required for startup, not preserve broad config document shape.

### 3. Delivery mechanism separate from authority model
Whether the child consumes:
- an import-map file
- some stable generated artifact
- another mechanism

should remain separate from the conceptual authority model.

### 4. Synthetic authority as weakest class
Any generated artifacts must be treated as implementation residue unless proven otherwise.
They should never define the conceptual model.

---

## Distilled engineering conclusion
The deepest architectural flaw in the current bootstrap line is not merely temp files or random UUIDs.
It is this:

**the authority model is currently too flattened.**

Too many authority classes are being allowed to converge into one synthetic bootstrap world without a sharp explicit hierarchy.

That is why later redesign must not start from “how do we generate a better import-map?”
It must start from:

- what is primary truth?
- what is derived truth?
- what is interop-only?
- what is synthetic residue?

Once that ranking is clean, the implementation shape can follow.
