## Distilled question
Which mechanisms are standard/idiomatic integration patterns, and which are exceptional rescue patterns?

## Distilled answer
`u.bootstrap.ts` contains both legitimate driver-level integration mechanisms and exceptional Vite-8 rescue mechanisms.
The correct architectural move is **not** to reject the whole module wholesale.
It is to separate the durable integration seams from the emergency bootstrap fabrication seams.

## Classification standard used in this pass
For this note:

### Standard seam
A mechanism is classified as **standard** when it is:
- an understandable integration responsibility for a Vite/Deno driver
- deterministic or capable of becoming deterministic without changing its nature
- rooted in declared consumer/workspace truth rather than ad hoc runtime fabrication
- plausible as long-lived architecture

### Exceptional seam
A mechanism is classified as **exceptional** when it is:
- a workaround for a specific toolchain/runtime mismatch
- dependent on synthetic artifacts or runtime fabrication
- shaped by rescue pressure rather than by principled top-down factoring
- something that should be challenged before becoming permanent architecture

### Borderline seam
A mechanism is classified as **borderline** when it is:
- probably justified at this layer
- but currently expressed in a rescue-shaped or over-broad form
- likely to remain in some form, but not necessarily in its current implementation

## Standard seams

### 1. Version-gated bootstrap activation
**Observed shape:**
- only activate this path for Vite 8+

**Classification:**
- standard

**Why:**
A compatibility driver needs a version gate when upstream behavior changed materially.
The existence of a version gate is normal engineering, not scar tissue.

**Keep principle:**
- preserve the concept
- exact branching logic can still be refined

---

### 2. Consumer/workspace Deno authority discovery
**Observed shape:**
- nearest `deno.json`
- workspace-root `deno.json`
- import-map reference discovery

**Classification:**
- standard

**Why:**
A Deno-aware Vite driver legitimately needs to locate consumer/workspace authority.
This is core integration work, not a workaround.

**Keep principle:**
- preserve the role
- challenge only breadth and merge strategy later

---

### 3. Import authority extraction and normalization
**Observed shape:**
- extract `imports`
- read import-map file
- normalize to string-records

**Classification:**
- standard

**Why:**
Projecting declared import authority into a form usable by a runtime/tool is an ordinary integration seam.
This is close in spirit to the cleaner `m.vite.config.workspace` alias projection seam.

**Keep principle:**
- preserve the role
- later passes should ask how narrow/minimal the extraction should be

---

### 4. Canonical ordering of synthesized imports
**Observed shape:**
- `sortImports(...)`

**Classification:**
- standard

**Why:**
Canonicalization for stable output is good long-lived engineering.
This is clearly not rescue logic.

**Keep principle:**
- preserve and strengthen

---

### 5. Separation of bootstrap authority from app/plugin resolution
**Observed shape:**
- explicit comment and design posture that bootstrap is its own seam

**Classification:**
- standard

**Why:**
This is one of the strongest durable design choices in the whole line.
`@deno/vite-plugin` operates inside plugin resolution/load; `u.bootstrap.ts` exists because config/bootstrap is a different seam.
Recognizing that separation is standard and correct.

**Keep principle:**
- non-negotiable unless an equally proven replacement seam exists

---

## Borderline seams

### 6. Root + nearest authority merge
**Observed shape:**
- merge root imports/json with nearest imports/json

**Classification:**
- borderline

**Why:**
Some kind of authority composition may be required in a workspace-aware driver.
That part is believable.
What is not yet obviously standard is the **current broad merge shape**.

**Engineering read:**
- the responsibility is standard
- the current expression is not obviously final

**Later question:**
- what is the minimum truthful composition model?

---

### 7. Toolchain runtime import injection
**Observed shape:**
- inject `vite`, `vite/internal`, `vite/module-runner`, node builtins

**Classification:**
- borderline

**Why:**
A bootstrap seam may legitimately need to supply toolchain runtime authority.
That is believable for a Vite/Deno driver.
But the current hand-assembled import-map injection is rescue-shaped.

**Engineering read:**
- likely a real owned seam
- current implementation still feels workaround-heavy

---

### 8. Explicit transitive bootstrap dependency declaration
**Observed shape:**
- hard-coded bootstrap deps such as `rolldown`, `postcss`, `lightningcss`

**Classification:**
- borderline moving toward exceptional

**Why:**
There are cases where bootstrap/runtime closure must be made explicit.
But pinning a transitive closure manually is not ideal long-term shape unless it is truly the only stable contract.

**Engineering read:**
- may be an earned seam
- current form should be treated as highly challengeable

---

### 9. Conditional consumer-sensitive plugin runtime extension
**Observed shape:**
- check `package.json` for `@vitejs/plugin-react`
- inject additional runtime dep if present

**Classification:**
- borderline moving toward exceptional

**Why:**
Adapting bootstrap to consumer capabilities is understandable.
But probing Node metadata to decide import-map authority in a Deno-first seam is not obviously the clean final model.

**Engineering read:**
- the adaptability goal is standard
- the current trigger/detection path is likely not the final expression

---

## Exceptional seams

### 10. Random temp bootstrap import-map file generation
**Observed shape:**
- `.vite.bootstrap.${uuid}.imports.json`

**Classification:**
- exceptional

**Why:**
This is a direct rescue mechanism.
It creates a synthetic bootstrap artifact with unstable identity.
This is not what long-lived clean integration should aim to look like.

**Engineering read:**
- treat as a provisional compatibility maneuver
- challenge hard in refactor

---

### 11. Random temp bootstrap module file generation
**Observed shape:**
- `.vite.bootstrap.${uuid}.module-sync-enabled.mjs`

**Classification:**
- exceptional

**Why:**
Same reasoning as above, even more strongly.
This is a generated physical shim module inserted to satisfy a runtime seam.
It may have been necessary during the probe line, but it is rescue architecture by default.

**Engineering read:**
- provisional seam only
- must justify itself again in top-down replacement

---

### 12. Ephemeral artifact lifecycle with cleanup callback
**Observed shape:**
- create generated files
- hand back cleanup function
- remove them afterward

**Classification:**
- exceptional

**Why:**
Temporary artifact lifecycle management is what follows from choosing transient generated state as the solution.
This is not inherently shameful, but it is not the shape to aim for unless proven unavoidable.

**Engineering read:**
- derivative of the temp-file strategy
- should shrink or disappear if the strategy changes

---

### 13. Package.json anchor probing inside this bootstrap seam
**Observed shape:**
- walk upward to find `package.json`
- read dependencies/devDependencies for plugin detection

**Classification:**
- exceptional

**Why:**
In this exact seam, this reads as an interop compromise rather than a clean primary authority model.
It may be expedient, but it is not the most principled expression of Deno/Vite bootstrap truth.

**Engineering read:**
- challenge strongly
- preserve only if no cleaner authority source exists

---

## Clean contrast from adjacent modules

### `m.vite.config.workspace`
This module is overwhelmingly **standard** in its shape.
It:
- discovers workspace packages
- reads declared exports
- projects aliases deterministically

It feels like long-lived architecture.
That makes it a useful north star for what “standard” looks like in this package.

### `@deno/vite-plugin`
That plugin also feels mostly **standard** within its own intended seam.
It works in Vite’s plugin pipeline and explicitly declares that config/bootstrap is out of scope.
Its limitation is not architectural failure; it is a boundary declaration.

That comparison matters:
- `@deno/vite-plugin` did not fail by being bad
- `u.bootstrap.ts` exists because the decisive seam lives outside the plugin’s intended responsibility

---

## Classification summary

### Standard and likely durable
- version-gated activation
- Deno/workspace authority discovery
- import authority extraction and normalization
- canonical ordering
- keeping bootstrap separate from app/plugin resolution

### Borderline and likely to survive only in refactored form
- root + nearest authority composition
- toolchain runtime import injection
- explicit bootstrap dependency declaration
- consumer-sensitive runtime extension

### Exceptional and should be treated as rescue-shaped until re-proven
- random temp bootstrap import-map files
- random temp shim module files
- cleanup-driven ephemeral artifact lifecycle
- package.json probing as a bootstrap authority trigger in this seam

---

## Distilled architectural conclusion
The right move is **not** to abolish the bootstrap seam.
The right move is to:

1. preserve the standard seams
2. re-factor the borderline seams into cleaner owned contracts
3. aggressively challenge the exceptional seams and justify each one from first principles

That is the clean separation between:
- durable driver-vite architecture
and
- probe-line rescue residue
