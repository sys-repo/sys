## Distilled question
What looks like accumulated survival logic rather than principled shape?

## Distilled answer
The scar tissue in `u.bootstrap.ts` is not the existence of a bootstrap seam.
The scar tissue is the set of **probe-line survival moves** that fabricate a synthetic child-runtime world with unstable identity and over-broad authority just to get Vite 8 through the bootstrap frontier.

In short:
- **bootstrap as a seam is not scar tissue**
- **the current rescue-shaped fabrication inside that seam is scar tissue**

---

## What counts as scar tissue in this pass
For this note, something counts as scar tissue when it has one or more of these traits:

- it exists because the blocker had to be moved forward under pressure
- it manufactures or patches runtime reality instead of projecting declared truth cleanly
- it introduces unstable identity or transient artifacts
- it broadens authority because precision was too expensive during the rescue line
- it solves the immediate breakage but leaves behind a shape we would not choose top-down

---

## High-confidence scar tissue

### 1. Random bootstrap import-map filenames
**Observed shape:**
- `.vite.bootstrap.${crypto.randomUUID()}.imports.json`

**Why this is scar tissue:**
This is the clearest rescue move in the file.
A random generated authority document in a consumer/package root is not a principled target shape.
It is a “make the child boot now” move.

**Why it matters:**
- unstable file identity
- likely worse cache reuse
- visible artifact leakage into package directories
- feels temporary because it is temporary

**Top-down read:**
If a bootstrap import-map artifact remains at all, it should not survive in this random form.

---

### 2. Random bootstrap shim module filenames
**Observed shape:**
- `.vite.bootstrap.${crypto.randomUUID()}.module-sync-enabled.mjs`

**Why this is scar tissue:**
This is another direct runtime patch artifact.
A generated shim module with random identity is rescue-shaped by default.

**Why it matters:**
- unstable file URL identity
- likely anti-cache
- another visible artifact in consumer roots
- encodes “we needed to satisfy one runtime seam somehow” rather than “this is the right model”

**Top-down read:**
This should be assumed removable unless re-earned from first principles.

---

### 3. Cleanup-driven ephemeral artifact lifecycle
**Observed shape:**
- create temp files
- return `cleanup()` to remove them

**Why this is scar tissue:**
This is the natural follow-on from the temp-file strategy.
Once the seam depends on transient generated artifacts, cleanup becomes necessary.
That makes cleanup itself derivative scar tissue.

**Why it matters:**
- the seam becomes operationally fragile
- lifecycle complexity appears only because the artifact model is already rescue-shaped
- it encodes “leave no trace after the workaround” rather than “use a stable legitimate runtime contract”

**Top-down read:**
If the artifact strategy changes, much of this should disappear with it.

---

### 4. Whole-document carry-through of import-map JSON
**Observed shape:**
- `json: { ...(root?.json ?? {}), ...(nearest?.json ?? {}) }`
- writing `{ ...source.json, imports }`

**Why this is scar tissue:**
This reads like “preserve the document shape we found so we do not break unknown cases” rather than “project the minimal exact authority contract.”
It is broad defensive carry-over.

**Why it matters:**
- broadens bootstrap payload unnecessarily
- increases the chance of accidental inherited behavior
- hides the true minimum contract behind document preservation

**Top-down read:**
This is a classic sign that the seam was made to work before its contract was fully distilled.

---

### 5. Over-broad root + nearest authority merge
**Observed shape:**
- merge root imports/json with nearest imports/json

**Why this is scar tissue:**
Some authority composition may be legitimate.
The scar tissue is the **broad merge-first posture**.
It looks like the rescue line chose to union authority to make failures disappear rather than to prove the narrow required model.

**Why it matters:**
- synthetic runtime worldview
- harder to reason about actual truth source
- easier to carry hidden authority than to model precise authority

**Top-down read:**
The merge may survive, but almost certainly not in this broad “carry everything forward” form.

---

### 6. Hard-coded bootstrap dependency closure lists
**Observed shape:**
- `VITE_BOOTSTRAP_DEPENDENCIES`
- `ROLLDOWN_BOOTSTRAP_SUBPATHS`
- `PLUGIN_REACT_BOOTSTRAP_DEPENDENCIES`

**Why this is scar tissue:**
These lists read like a record of blockers encountered and then pinned through manually.
They are effectively a compatibility excavation log embedded in the runtime seam.

**Why it matters:**
- maintenance gravity
- upgrade fragility
- runtime contract not obvious from first principles
- close to “append the next missing thing when it breaks” architecture

**Top-down read:**
These lists may reflect some real owned seam, but in current form they strongly smell like blocker-driven accumulation.

---

### 7. `package.json` probing as bootstrap trigger logic
**Observed shape:**
- `packageAnchor(...)`
- `consumerVersion(...)`
- conditional runtime extension based on `@vitejs/plugin-react`

**Why this is scar tissue:**
This reads like a targeted pragmatic patch, not a clean primary authority model.
In a Deno/workspace/bootstrap seam, this is a “check one more ambient thing so the current case works” move.

**Why it matters:**
- mixes Node metadata into a Deno bootstrap seam
- makes bootstrap behavior depend on upward package.json probing
- shifts the seam away from explicit declared authority

**Top-down read:**
This may have been justified during the probe, but it is not the shape to carry forward casually.

## Medium-confidence scar tissue

### 8. Bootstrap import-map synthesis itself, in the current line
**Observed shape:**
- generate a fresh import-map specifically for the child runtime

**Why this is scar tissue in current form:**
This note does **not** assume every possible generated bootstrap map is automatically wrong.
But in the current line, this mechanism is part of the rescue stack and is explicitly being treated as a hack unless re-proven.

**Why it matters:**
- it embodies the current synthetic runtime strategy
- it is the carrier for multiple other scar-tissue choices

**Top-down read:**
Current posture remains correct:
- assume this should be dropped unless re-earned by a fully STIER/TMIND design

---

### 9. Bootstrap shim indirection via `#module-sync-enabled`
**Observed shape:**
- generated module file
- mapped through import authority

**Why this is scar tissue in current form:**
The abstract need may be real.
The current route—physical temp shim file + mapped file URL—looks like a direct compatibility patch more than a final owned contract.

**Top-down read:**
Treat current implementation as rescue residue unless later passes prove otherwise.

## What is **not** scar tissue

### 1. Version gate for Vite 8+
This is standard compatibility engineering.
It is not scar tissue just because it is conditional.

### 2. Deno/workspace authority discovery
Finding relevant authority anchors is normal driver work.

### 3. Import extraction and normalization
Projecting declared imports into usable form is normal integration work.

### 4. Sorting synthesized imports
Canonicalization is good engineering, not residue.

### 5. Keeping bootstrap separate from app/plugin resolution
This is one of the cleanest design truths in the line.
It should survive.

## Scar-tissue pattern summary
The scar tissue clusters into four larger patterns:

### A. Transient artifact pattern
- random import-map file
- random shim module file
- cleanup lifecycle

This is the strongest residue cluster.

### B. Broad-carryover pattern
- whole JSON carry-through
- broad root + nearest merge

This indicates precision was deferred during the rescue line.

### C. Manual closure pattern
- hard-coded dependency lists
- subpath lists
- optional plugin-specific bootstrap additions

This indicates blocker-driven accumulation rather than distilled contract.

### D. Ambient fallback pattern
- package.json anchor probing
- dependency-based runtime branching

This indicates the seam was widened opportunistically to satisfy runtime needs.

## Distilled engineering conclusion
The most important scar tissue is not “ugly code style.”
It is **unstable fabricated runtime state**.

That is the real residue to remove.
Everything else is secondary to that.

If the redesign does only one thing right, it should eliminate the pattern of:
- random bootstrap artifacts
- broad synthetic carry-over
- blocker-driven runtime closure patching

and replace it with:
- stable authority identity
- explicit minimum contract
- clearly owned integration hooks
