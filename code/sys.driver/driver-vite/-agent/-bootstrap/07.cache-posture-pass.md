## Distilled question
What parts of this design support cache reuse, and what parts destroy it?

## Distilled answer
The current bootstrap seam has **some cache-friendly content behavior**, but its delivery model is structurally hostile to cache reuse.

The most important fact is:

**the seam currently optimizes canonical payload content while making the runtime handles that carry that content look new every run.**

That is poor cache posture.

So the current line is not “uncached because nothing is stable.”
It is worse in a more specific way:
- the seam stabilizes secondary things
- while destabilizing primary reuse surfaces

---

## What cache posture means in this pass
For this note, cache posture is not limited to one specific cache implementation.
It means:

- does this seam make equivalent work recognizable as equivalent?
- does it preserve stable handles that upstream/downstream tools could reuse?
- does it avoid forcing fresh bootstrap identity for semantically unchanged runs?

This includes, in principle:
- Deno module/runtime caching
- Vite startup/config reuse behavior
- any command/input keyed reuse
- filesystem artifact reuse
- human/debugger ability to recognize “same run shape”

---

## Cache-friendly surfaces already present
These are the parts of the seam that currently help cacheability in principle.

## 1. Import entry sorting
**Observed shape:**
- `sortImports(...)`

**Why it helps:**
Canonical ordering reduces meaningless diff churn in generated content.
That is a real cache-friendly behavior.

**Limit:**
It only helps content stability, not artifact identity.

---

## 2. Deterministic config discovery and normalization
**Observed shape:**
- nearest/root lookup
- normalized string-record imports

**Why it helps:**
Equivalent consumer/workspace state produces equivalent extracted authority content.
That means the seam is at least capable of stable semantic output.

**Limit:**
Again, this helps content, not delivery identity.

---

## 3. Stable Vite specifier derivation
**Observed shape:**
- `npm:vite@<resolved version>`

**Why it helps:**
For a fixed dependency state, the child runtime target is stable.
That gives the command line one meaningful stable anchor.

**Limit:**
The generated import-map path still causes overall command identity churn.

---

## 4. Existing Vite cache awareness in adjacent command logic
**Observed shape in `u.wrangle.ts`:**
- explicit `viteCacheDir(...)`
- write roots include `.vite`

**Why it helps:**
This shows the surrounding driver already knows cache directories matter and is trying to preserve them operationally.

**Limit:**
The bootstrap seam currently undercuts this by making startup identity itself unstable.

---

## Cache-hostile surfaces
These are the decisive cache-destroying choices in the current design.

## 1. Random import-map artifact path
**Observed shape:**
- `.vite.bootstrap.${uuid}.imports.json`

**Why it hurts cache posture:**
Equivalent bootstrap payloads are delivered under new paths every run.
Anything sensitive to exact import-map path identity sees a new input every time.

**Why this matters:**
For caching, path identity is not superficial.
A different authority handle is often enough to force conservative re-evaluation.

**Cache posture classification:**
- high-severity anti-cache design

---

## 2. Random shim module file URL
**Observed shape:**
- generated `.module-sync-enabled.mjs`
- imported through a new file URL each run

**Why it hurts cache posture:**
This changes one of the actual specifier values inside the projected authority map.
So the bootstrap payload is not just delivered under a new wrapper path; part of its import surface is also freshly named.

**Why this matters:**
This is worse than path churn alone because it mutates an internal runtime import handle.
For any loader/module system that keys behavior off URL/specifier identity, this looks like a genuinely new module.

**Cache posture classification:**
- extremely high-severity anti-cache design

---

## 3. Ephemeral cleanup lifecycle
**Observed shape:**
- create temp artifacts
- remove them after use

**Why it hurts cache posture:**
A cache-friendly seam tends toward stable reusable handles or stable derivable artifacts.
This seam instead chooses disposable artifacts whose very lifecycle guarantees replacement on next run.

**Why this matters:**
It means the implementation is culturally and operationally optimized for:
- one-shot validity
not
- repeatable reuse

**Cache posture classification:**
- foundational anti-cache posture

---

## 4. Command-line identity churn
**Observed shape:**
- `--import-map=<new path each run>`

**Why it hurts cache posture:**
Even if the child command is semantically “the same Vite launch,” it is operationally different each time.
That reduces the chance of stable recognition at the command/startup boundary.

**Why this matters:**
A cache cannot reuse what it cannot cheaply recognize as equivalent.

**Cache posture classification:**
- high-severity anti-cache design

---

## 5. Synthetic bootstrap artifact as central delivery mechanism
**Observed shape:**
- merged authority must be materialized into generated files before child startup

**Why it hurts cache posture:**
When generated artifacts are the central delivery mechanism, cache quality depends heavily on artifact identity discipline.
The current line lacks that discipline.

**Why this matters:**
This means the implementation is paying the cost of an artifact model **without** claiming the benefits of stable artifact reuse.

**Cache posture classification:**
- medium-to-high structural cache weakness

---

## The current cache split
This seam should be understood as having two very different cache layers.

### Layer A — content-level cache friendliness
These parts are relatively good:
- stable discovery
- stable normalization
- sorted imports
- stable Vite version selection

### Layer B — runtime-handle cache friendliness
These parts are bad:
- random artifact paths
- random file URLs
- disposable lifecycle
- command identity churn

This is the critical insight.

**The seam is cache-friendly at the wrong layer.**

It is making the payload neat while making the runtime handle disposable.
That is not enough.

---

## Equivalent-run cache thought experiment
Take two runs with all meaningful semantic inputs unchanged.

### Stable in principle
- same consumer authority
- same workspace authority
- same Vite version
- same bootstrap dependency lists
- same startup contract

### What still changes now
- import-map filename
- shim filename
- shim file URL
- command-line import-map arg
- artifact existence/lifecycle

### Result
A reuse-oriented system looking at runtime handles sees:
- new startup inputs
- new artifact identities
- new module URLs

So even though the *meaning* is the same, the *cache-facing surface* looks changed.

That is the core cache-posture failure.

## Why this likely shows up as “still feels cold”
This note should make that feeling legible.

If the seam constantly changes:
- import-map path identity
- internal shim URL identity
- command startup identity

then a human can easily observe:
- second run still feels too fresh
- startup feels less reused than it should
- “we improved structure but not the coldness”

That feeling is not mystical.
It is structurally plausible from the current cache posture.

## Contrast with better cache posture examples

## `m.vite.config.workspace`
This module is far more cache-friendly in shape because:
- it projects deterministic aliases
- it does not create transient artifact identities
- equivalent workspace state produces equivalent projected output identity

That is a much healthier cache posture.

## `@deno/vite-plugin`
Even though it does not solve your seam, it also shows a stronger cache posture tendency:
- per-environment caches
- loader reuse
- config discovery without central random artifact generation

Again, this does not solve config/bootstrap.
But it proves the issue here is not “all Deno/Vite integration must feel uncached.”

## What the essential contract says about cache posture
From pass 04, the essential contract only requires:
- minimum truthful startup authority for the child

It does **not** require:
- disposable delivery artifacts
- random identity
- startup handle churn

So the current cache weakness is not contract-bound.
It is implementation-bound.

This is a very important conclusion.
It means cache posture can be improved **without** violating the essential seam role.

---

## Cache posture requirements for the replacement shape
This pass implies the replacement must satisfy:

### 1. Stable delivery identity for equivalent runs
If the bootstrap authority is the same, the handle by which the child receives it must also be the same or derivably equivalent.

### 2. Minimal synthetic artifact dependence
If synthetic artifacts remain, they must be:
- stable-path
or
- content-addressed
or
- otherwise explicitly reuse-oriented

### 3. Runtime module handles must stop churning casually
Especially file-URL-backed injected modules.
If a shim remains, its identity must not change without semantic cause.

### 4. Cache posture must be judged at the runtime boundary, not only at payload serialization level
This is the biggest conceptual correction.
It is not enough to sort keys in JSON if the import-map handle changes every run.

---

## Distilled engineering conclusion
The cache posture of the current line is mixed, but decisively flawed.

The best single-sentence summary is:

**the seam is cache-conscious in serialization, but cache-destructive in delivery.**

That is the wrong trade.

The replacement line must therefore treat:
- stable delivery identity
- stable module handle identity
- stable command identity

as first-class design goals, not afterthoughts.
