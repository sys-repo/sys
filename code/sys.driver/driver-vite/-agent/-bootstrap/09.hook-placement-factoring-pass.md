## Distilled question
At what layer should each responsibility actually live?

## Distilled answer
The current line works, but `u.bootstrap.ts` is still carrying **too many responsibilities in one place**, and `u.wrangle.ts` is carrying a few more than it should.

The clean placement is:

- **bootstrap projection** owns only startup authority projection
- **bootstrap delivery** owns how that projection becomes child-consumable
- **wrangle/command** owns child command assembly, permissions, env, and disposal wiring
- **workspace config projection** owns workspace alias/export projection for Vite config/app resolution
- **app/plugin resolution** owns ongoing module resolution after startup
- **public driver verbs** own stable package-facing `build()` / `dev()` entrypoints, not bootstrap internals
- **narrow interop / upstream delegation** owns only the explicitly justified pieces that must consult package metadata or defer to Vite native loader / `@deno/vite-plugin`

The sharpest placement correction is:

**bootstrap should stop being a mixed projection + artifact-fabrication + interop-detection + cleanup center.**

That is the core factoring result.

---

## Why this pass matters
Earlier passes established:
- the contract is small
- the authority model is currently too flat
- the cache posture is harmed by delivery identity churn
- the seam is truth-seeking but not yet truth-shaped

This pass answers the architectural follow-up:

**which module boundary should own each of those concerns?**

Without that answer, the later rewrite will drift into either:
- moving code around cosmetically
or
- recreating the same rescue shape in prettier files

---

## The current responsibility pile-up
Today, `u.bootstrap.ts` is doing all of these at once:

1. version-gating the seam (`viteMajor >= 8`)
2. discovering consumer/workspace Deno authority
3. flattening root + nearest authority
4. deciding bootstrap-owned Vite runtime imports
5. consulting consumer package metadata for conditional injection
6. fabricating synthetic delivery artifacts
7. assigning runtime identities to those artifacts
8. returning cleanup behavior

And `u.wrangle.ts` is doing all of these at once:

1. discovering Vite package authority
2. deciding config loader mode
3. deciding permissions
4. deciding env
5. deciding write roots / cache roots
6. invoking bootstrap creation
7. threading bootstrap delivery into command args
8. wiring disposal

This is workable in a probe line.
It is not the clean final factoring.

---

## Correct layer model
The clean layer model for the replacement should be:

### 1. Bootstrap projection layer
Owns:
- detect whether bootstrap handling is needed
- collect ranked startup authority inputs
- project the minimum startup authority model

Does **not** own:
- temp-file naming
- cleanup lifecycle as conceptual API
- broad package probing for unrelated runtime convenience
- app/plugin resolution

### 2. Bootstrap delivery layer
Owns:
- turning the projected authority model into whatever child-consumable handle is still needed
- stable identity strategy if any artifact remains
- disposal only as an implementation detail of delivery

Does **not** own:
- authority ranking itself
- workspace alias semantics
- public build/dev behavior

### 3. Wrangle / command layer
Owns:
- child `deno run` argument construction
- permission policy
- env setup
- config loader selection
- attaching optional bootstrap delivery handle
- composing cleanup/disposal from lower layers

Does **not** own:
- authority projection logic itself
- workspace alias synthesis
- transport id semantics

### 4. Workspace config projection layer
Owns:
- deterministic alias/export projection from workspace package truth into Vite config land
- package/workspace composition logic such as `m.vite.config.workspace`

Does **not** own:
- bootstrap-time Deno child authority delivery
- child process spawning

### 5. App/plugin resolution layer
Owns:
- the actual post-startup resolution behavior inside Vite/plugin/runtime land
- plugin-pipeline resolution
- dev/build transport shaping where appropriate

Does **not** own:
- bootstrap-time command shaping

### 6. Public driver verbs
Owns:
- `build(...)`
- `dev(...)`
- stable package-facing driver ergonomics

Does **not** own:
- exposing rescue bootstrap internals as public conceptual surface

### 7. Narrow interop / upstream delegation
Owns:
- explicit delegation boundaries to Vite native loader
- explicit delegation boundaries to `@deno/vite-plugin`
- sharply scoped package/runtime metadata checks only where they are truly needed

Does **not** own:
- broad local reimplementation of things upstream already handles correctly

**Correct home:**
- bootstrap projection layer

**Why:**
This is a bootstrap-contract question, not a command-formatting detail.

**Verdict:**
- keep near bootstrap
- but as seam activation, not mixed into artifact fabrication

## 2. Consumer/workspace Deno authority discovery
**Current location:**
- `u.bootstrap.ts`

**Correct home:**
- bootstrap projection layer

**Why:**
This is core truth-input discovery for startup authority.

**Verdict:**

**Current location:**

**Correct home:**
- bootstrap projection layer

**Why:**

**Verdict:**

## 4. Bootstrap-owned runtime import additions

## 5. Conditional interop decisions from `package.json`

## The most important factoring split
The replacement should enforce this explicit split:

### A. `Bootstrap.Project`
Pure-ish, ranked startup authority projection.
Input:
- cwd
- vite/runtime context
- consumer/workspace truth
- narrow interop facts

Output:
- startup authority model

### B. `Bootstrap.Deliver`
Child-consumable transport for that model.
Input:
- startup authority model

Output:
- optional handle for `deno run`
- optional cleanup/disposal only if still needed

### C. `Wrangle.command`
Command assembly over the above.
Input:
- paths
- subcommand
- env/permission/runtime facts
- optional bootstrap handle

Output:
- child args/env/dispose

This is the cleanest architecture lever in the whole pass.

---

## Why this split matters
If projection and delivery remain fused:
- authority ranking will stay muddy
- synthetic artifact choices will keep distorting truthfulness
- cache posture will keep being treated as a bootstrap-model issue instead of a delivery issue

If wrangle keeps too much interop/package logic mixed into bootstrap:
- command assembly and authority projection will stay mutually contaminating

If workspace alias projection is collapsed back into bootstrap:
- startup seam and runtime/app seam will blur again

So this factoring is not aesthetic.
It is the structural precondition for the final clean replacement.

---

## Placement mistakes to avoid in the rewrite

### 1. Do not move workspace alias logic into bootstrap
That would re-collapse startup and app/runtime resolution.

### 2. Do not move permissions/env into bootstrap
That would confuse child launch policy with authority projection.

### 3. Do not expose delivery artifacts as the public API story
Users want `build()` / `dev()`, not “here is our bootstrap temp file abstraction.”

### 4. Do not let package.json probing masquerade as declared truth
Keep it explicit and secondary.

### 5. Do not push bootstrap responsibility into app/plugin resolution just because upstream plugins exist
`@deno/vite-plugin` does not solve `vite.config.ts` bootstrap.
That boundary remains yours.

---

## Distilled engineering conclusion
The current line already contains the right major seams, but one of them is still internally over-merged.

The real factoring correction is:

**bootstrap must be split into projection and delivery, while wrangle remains the command seam and workspace config projection remains a separate authority-projection seam.**

That is the proper placement because it is the shortest path to a bootstrap seam that is:
- minimal
- truthful
- deterministic
- published-boundary-correct
- still anchored to the proven Vite 8 line

Everything else in this pass is downstream of that one move.
