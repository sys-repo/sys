## Distilled question
What mechanisms are actually being used here?

## Distilled answer
In its current form, `u.bootstrap.ts` behaves like a **bootstrap authority synthesizer** for the Vite 8 child runtime.
It is not one mechanism; it is a stack of mechanisms combined into one bootstrap seam.

## Grouped mechanism families

### A. Authority-location mechanisms
These answer: where does truth come from?
- nearest `deno.json`
- workspace-root `deno.json`
- referenced import-map file
- nearest `package.json`

### B. Authority-shaping mechanisms
These answer: how is truth transformed?
- import extraction
- string-record normalization
- root + nearest merge
- sorted import serialization

### C. Runtime-bootstrap mechanisms
These answer: what must be injected so the child can start?
- generated bootstrap import map
- generated module-sync shim
- Node builtin aliases
- Vite and rolldown runtime imports
- pinned transitive runtime deps

### D. Consumer-adaptive mechanisms
These answer: how does bootstrap vary with the consuming app?
- package.json dependency probing
- plugin-react-sensitive bootstrap extension

### E. Lifecycle mechanisms
These answer: how is generated state managed?
- temp file creation
- cleanup callback

## Relationship to adjacent clean shapes

### `m.vite.config.workspace`
That module is using a much narrower mechanism family:
- workspace discovery
- package export extraction
- alias projection

It is a useful contrast because it is performing **deterministic authority projection** rather than **synthetic bootstrap fabrication**.

### `@deno/vite-plugin`
That plugin mainly uses:
- Vite `resolveId`
- Vite `load`
- Deno loader-backed specifier resolution
- remote/module rewrite support inside the plugin pipeline

It does **not** primarily use the generated-bootstrap-artifact pattern found in `u.bootstrap.ts`.
That reinforces that `u.bootstrap.ts` is solving a different seam: child bootstrap/config/runtime startup.

---

## Mechanism taxonomy summary
At mechanism level, `u.bootstrap.ts` is best described as a composition of:

- authority discovery
- authority extraction
- authority merge
- bootstrap-specific import-map synthesis
- temporary module artifact injection
- runtime dependency injection
- consumer-sensitive bootstrap branching
- ephemeral artifact lifecycle management

The file is therefore not a simple helper.
In its current form, it is a compact **bootstrap authority construction stack**.
