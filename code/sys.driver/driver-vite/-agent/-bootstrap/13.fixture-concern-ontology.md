## Purpose
This note is **not** a file index.
It is a distillation of:
- what kinds of fixture worlds the test suite has needed
- what proof burdens those fixture worlds were carrying
- what the higher-order fixture concerns really are beneath the concrete folders and helpers
- what a more principled fixture model should look like for the bootstrap rewrite and its proof line

## Distilled question
What kinds of consumer worlds, authority worlds, and runtime boundaries do the current tests need to construct, and what are the irreducible fixture concerns beneath those concrete fixtures?

## Distilled answer
The current `driver-vite` tests are not mainly asking for “sample apps.”
They are staging **proof worlds**.

That is the key read.

A fixture in this package is usually doing one or more of these things:
- staging a consumer boundary
- staging an authority boundary
- staging a launch-phase boundary
- staging an interop boundary
- staging an output/runtime-observable boundary

So the right index is not:
- fixture name → folder path

The right index is:
- fixture world → concern bundle → proof obligation

That is the durable model.

## What the current fixtures are doing
Looking across the current `src/-test`, `src/m.vite/-test`, `src/m.vite/-test.external`, `src/m.vite.transport/-test`, `src/m.vite.config/-test`, and related helpers, the suite is constructing several recurring fixture world families.

## 1. Minimal local consumer worlds
Representative surfaces:
- `src/-test/vite.sample-1`
- `src/-test/vite.sample-2`
- `src/-test/vite.sample-3`
- `src/-test/vite.sample-std-path`

### What these worlds are proving
- local Vite app startup shape
- basic config evaluation path
- basic build/dev lifecycle correctness
- simple entry/outDir/base routing

### What they are **not** primarily proving
- published boundary truth
- sophisticated authority ranking
- bootstrap residue legitimacy

### Distilled essence
These are **baseline viability worlds**.
They answer:
- does the lane work at all under a small local consumer?

---

## 2. Bridge / authority-injection worlds
Representative surfaces:
- `src/-test/vite.sample-bridge`
- `src/m.vite/-test/u.bridge.fixture.ts`
- `src/m.vite/-test/-bridge.integration.test.ts`

### What these worlds are proving
- local bootstrap/import authority can be injected or reconstructed well enough for build/dev to work
- local `@sys/*` imports can be translated into a consumer-usable fixture world
- bridge-time rewrite/injection can be made truthful enough for startup

### Important observation
`u.bridge.fixture.ts` is doing much more than “copying a sample.”
It:
- computes root package version authority
- computes root import-map authority
- inspects reachable package specifiers
- bridges `@sys/*` package imports to workspace paths or JSR/npm authority
- patches fixture `imports.json`, `deno.json`, `package.json`, and even `vite.config.ts`

That means this helper is really a **fixture world synthesizer**.

### Distilled essence
These are **authority-bridge worlds**.
They answer:
- can the test construct a truthful enough consumer world for bootstrap-boundary proof?

## The higher-order fixture concerns
These are the fixture concerns beneath the concrete folders.
This is the most important part of the note.

A principled fixture system for this package should model concerns like these explicitly.

## A. Consumer-boundary concern
What kind of consumer is being simulated?

### Values
- local package consumer
- workspace consumer
- copied fixture consumer
- published/package-boundary consumer
- generated/external repo consumer

### Why it matters
This is one of the most important proof axes in the whole package.
A local mono-repo world and a published consumer world are not interchangeable.

---

## B. Authority concern
What authority world is being exercised?

### Values
- nearest `deno.json`
- import-map file
- inline imports
- workspace-root authority
- root + nearest interaction
- local bridge authority
- runtime-added authority

### Why it matters
Most bootstrap design mistakes are really authority-shape mistakes.
Fixtures should make that visible.

---

## C. Launch-phase concern
What phase is under proof?

### Values
- pre-plugin bootstrap
- config discovery
- config evaluation
- build startup
- dev startup
- post-startup resolution
- production output verification

### Why it matters
The same consumer world can need different fixtures for different phases.
The package's decisive seam is phase-sensitive.

---

## D. Interop concern
What non-Deno boundary is in play?

### Values
- npm package metadata
- Vite version discovery
- esbuild runtime
- plugin-react presence
- rolldown subpath/bootstrap deps

### Why it matters
Interop facts are often necessary, but they are not primary truth.
Fixtures should expose them as interop concerns, not let them blur into authority truth.

---

## E. Path-root concern
What root geometry is being stressed?

### Values
- normal root
- copied fixture root
- nested temp root
- different caller cwd vs config cwd
- workspace child path projection

### Why it matters
A surprising amount of real seam truth lives in root/path geometry.
The current suite already knows this.
The fixture ontology should say it directly.

---

## F. Delivery concern
How is startup authority delivered to the child?

### Values
- import-map handle present
- generated artifact required
- cleanup/disposal involved
- stable vs unstable handle identity

### Why it matters
This concern is present, but it is not yet explicit enough in the proof surface.
The rewrite should make it first-class.

---

## G. Output-observable concern
What observable behavior is being asserted?

### Values
- child starts
- dev becomes ready
- build emits artifacts
- built HTML contains expected entry
- built JS contains or omits specific markers
- no dev transport IDs leak
- elapsed/lifecycle behavior is correct

### Why it matters
A fixture without a crisp observable is often just scenery.
This concern keeps proof honest.

---

## H. Truthfulness concern
What truth claim is under proof?

### Values
- consumer truth preserved
- workspace derivation remains honest
- published boundary remains honest
- bootstrap additions remain subordinate
- local bridge does not become a fake final answer

### Why it matters
This is where architecture and proof finally meet.
Truthfulness is not just a design goal; it should be a fixture concern.

## I. Determinism/cache concern
What repeated-run identity or reuse claim is being exercised?

### Values
- equivalent-run artifact identity
- stable command identity
- no visible consumer-root residue
- cache-friendly repeated startup

### Why it matters
This is the fixture concern most obviously underrepresented right now.
Earlier architecture passes made it central.
The fixture system should prove it directly.

## The current fixture ontology in compact form
A concrete fixture can be understood as a tuple like:

- `consumer`
- `authority`
- `phase`
- `interop?`
- `pathRoot?`
- `delivery?`
- `observable`
- `truthfulness?`
- `determinism?`

That is much more durable than a folder name.

Example in spirit:
- consumer: `published`
- authority: `nearest + importMap`
- phase: `build`
- interop: `transitive jsr + npm runtime`
- delivery: `bootstrap handle required`
- observable: `build succeeds and no dev transport ids leak`
- truthfulness: `published consumer world remains honest`

That is the level of abstraction the rewrite should reason about.

## What the current suite is carrying as historical fixture residue
These are not accusations. They are the likely residue patterns visible from the current proof surface.

## 1. Some fixtures are concrete-world-heavy rather than concern-first
The suite has several named sample worlds whose deeper concern overlap is not yet explicit.
That is normal for a lived-in test line.

## 2. Bridge helpers carry a lot of authority synthesis at once
`u.bridge.fixture.ts` is clearly valuable, but it is also a sign that fixture synthesis has accumulated several concern classes into one helper:
- consumer authority
- package authority
- bridge authority
- toolchain authority
- config rewrite authority

That is useful proof machinery.
It is also a signal that a higher-order fixture model could become cleaner.

## 3. Some published fixtures include heavy pre-baked dependency state
For example the published UI baseline fixtures contain substantial `node_modules/.deno` state.
That may be justified for proof stability, but conceptually it means some fixtures are carrying:
- consumer world
- dependency store state
- package-manager/cache realism

all at once.

That is a real fixture concern bundle, not just “a sample app.”

## Missing or under-distilled fixture concerns
These are the higher-order proof concerns that seem worth strengthening explicitly.

## 1. Deterministic bootstrap identity worlds
The architecture distillation made this central.
The fixture line should eventually prove:
- equivalent startup authority produces equivalent delivery identity
- no random path churn for equivalent runs

## 2. No-visible-residue worlds
The definition of done strongly wants:
- no visible bootstrap debris in consumer/package roots

That should become a direct fixture concern, not merely an architectural wish.

## 3. Ranked-authority conflict worlds
The current suite proves that broad worlds work.
It should eventually prove the sharper question:
- what happens when nearest and root authority differ?
- what is the explicit ranking outcome?

## 4. Published consumer with minimal local crutches
The suite already has published-boundary worlds.
A principled rewrite should ensure some of those worlds are explicitly optimized around:
- least local privilege
- least bridge magic
- most boundary honesty

## 5. Startup handoff worlds
A final-shape seam should make it legible that:
- bootstrap owns early startup only
- upstream/runtime owns the rest

Fixtures could prove this more directly.

## What a more principled fixture system would look like
The final shape should likely move toward:

## 1. Concern-first fixture builders
Instead of only named sample folders, build helpers that can express concern bundles intentionally.

Example conceptual inputs:
- consumer mode
- authority mode
- phase
- interop needs
- path-root shape
- expected observables

## 2. Fewer accidental mega-fixtures
Some large concrete fixture trees should remain.
But they should be recognizable as:
- ecosystem realism fixtures
not
- the default way every concern gets tested

## 3. Explicit proof-world naming
Tests should increasingly say what world they need:
- published boundary
- root-shift config
- transitive jsr build
- bridge authority dev
- deterministic startup identity

That is better than mainly inheriting meaning from a fixture folder name.

## 4. Separation of fixture concerns from fixture transport
Just as bootstrap should separate projection from delivery, the fixture system should separate:
- proof world definition
from
- how the files are concretely realized on disk

That symmetry would actually make the test architecture much stronger.

## Distilled engineering conclusion
The fixture line in `driver-vite` is already doing serious work, but its deepest value is easier to see if you stop treating fixtures as sample folders and start treating them as **proof worlds**.

That is the main distillation.

The next refinement is:
- name fixture concerns explicitly
- separate concern bundles from concrete folder accidents
- make missing proof worlds legible, especially around:
  - deterministic startup identity
  - no bootstrap residue
  - ranked authority conflicts
  - published-boundary honesty

So the clean summary is:

**the current tests are not mainly indexing files; they are staging proof worlds. The right future fixture model should therefore be concern-first, world-first, and explicit about the proof burden each fixture carries.**
