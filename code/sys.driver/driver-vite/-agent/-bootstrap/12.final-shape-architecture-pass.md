## Distilled question
If this were designed cleanly from scratch today, what would the module boundaries and data flow be?

## Distilled answer
The correct final shape is **not** “a better temp import-map generator.”
It is:

**a narrow, ranked, deterministic bootstrap bridge that projects startup authority once, delivers it through a stable handle, launches the Vite child in the proven mode, and then delegates the rest to upstream/runtime seams.**

That is the architecture.

In one line:

**project truth, deliver stably, launch narrowly, delegate early.**

Everything in the final shape should serve that sentence.

---

## The target architecture in one picture

### Phase 1 — Discover
Collect only the startup-relevant facts:
- consumer Deno authority
- derived workspace authority only if truly inherited
- Vite runtime facts
- narrow npm/Vite interop facts

### Phase 2 — Project
Rank those facts into an explicit startup authority model:
- primary consumer truth
- derived workspace truth
- driver-owned startup additions
- interop-only adjustments

### Phase 3 — Deliver
Turn that model into a child-consumable handle with stable identity.
Prefer:
- no synthetic artifact

Otherwise accept only:
- deterministic or content-addressed driver-owned delivery
- outside consumer/package roots

### Phase 4 — Launch
Use wrangle to assemble the proven child command:
- `deno run`
- earned permissions/env
- `--configLoader=native`
- optional bootstrap handle

### Phase 5 — Delegate
Once the child is alive in the correct startup world:
- Vite native config loader owns config execution
- `@deno/vite-plugin` / upstream loader seams own in-pipeline Deno resolution
- app/plugin/transport seams own everything after startup

That is the full system.

## The final module boundary set
This is the clean top-down boundary set the package should converge to.

## 1. `Bootstrap.Project`
### Purpose
Project the minimum truthful startup authority model for the Vite 8 child.

### Owns
- Vite-8 bootstrap seam activation
- ranked authority discovery
- projection of startup-relevant imports/authority
- explicit separation of:
  - consumer truth
  - derived workspace truth
  - driver-owned startup additions
  - interop-only inputs

### Does not own
- temp-file fabrication
- cleanup as conceptual API
- child command assembly
- app/plugin resolution
- transport semantics

### Why it exists
This is the actual heart of the seam.
Not file generation.
Not cleanup.
Not command formatting.

### Design posture
This layer should feel like `m.vite.config.workspace` in spirit:
- narrow
- deterministic
- projection-oriented
- easy to reason about

---

## 2. `Bootstrap.Deliver`
### Purpose
Convert the projected startup authority model into whatever child-consumable handle is still required.

### Owns
- delivery format selection
- stable identity strategy
- optional materialization
- optional cleanup only as a delivery detail

### Does not own
- authority ranking
- truth modeling
- public API semantics
- package boundary meaning

### Ideal endpoint
- no synthetic artifact at all

### Acceptable endpoint if an artifact remains
- deterministic or content-addressed
- driver-owned location
- not written into consumer/package roots as visible residue
- stable across equivalent semantic runs

### Most important rule
Delivery must never define the conceptual model.
It is transport only.

---

## 3. `Wrangle.command`
### Purpose
Own the child launch contract.

### Owns
- `deno run` argument construction
- config-loader mode selection
- permission policy
- env wiring
- attaching the optional bootstrap handle
- composing disposal from delivery if still needed

### Does not own
- authority projection policy
- broad config discovery policy
- app/runtime resolution policy

### Why this stays
This is already the right seam.
It just needs cleaner inputs from bootstrap.

---

## 4. Narrow interop helpers
### Purpose
Answer the few runtime-package questions that are real npm/Vite interop facts rather than bootstrap truth.

### Owns
- Vite package/specifier discovery
- esbuild package/runtime facts for launch
- narrowly justified package-metadata checks

### Does not own
- consumer authority truth
- workspace truth hierarchy
- bootstrap delivery mechanics

### Why this matters
Interop facts are real.
But they are not primary truth.
They must stay visibly secondary.

---

## 5. `m.vite.config.workspace`
### Purpose
Remain the separate workspace alias/export projection seam.

### Owns
- workspace/package export truth projection into Vite config/app composition

### Does not own
- child bootstrap authority
- command launch
- startup delivery

### Why it stays separate
It is not bootstrap.
It is a different product from some shared source truth.
That separation is healthy and should remain.

---

## 6. Upstream/runtime-owned seams
### Vite native config loader owns
- config execution semantics
- native config loading behavior

### `@deno/vite-plugin` / upstream loader owns
- in-pipeline Deno resolution/load behavior
- `npm:` / `jsr:` / `http:` / `https:` handling after startup

### App/plugin/transport seams own
- everything that happens after startup authority has done its job

### Final doctrine
Own the early bridge.
Delegate the rest.

## The final data model
The final shape should be driven by a clean data model, even if the code names differ.

## A. Startup authority inputs
These are inputs, not yet output form.

### Consumer truth
- nearest consumer-relevant Deno authority
- startup-relevant imports only

### Derived workspace truth
- only what the consumer truth actually inherits or requires
- not broad workspace carry-through by default

### Driver-owned startup additions
- the minimum Vite child-runtime additions truly required before upstream seams can help

### Interop facts
- package/runtime facts needed only for justified npm/Vite boundary handling

---

## B. Startup authority model
The projected model should preserve ranking explicitly.
It should not begin life as a flattened synthetic import-map blob.

Conceptually:

- `consumer`
- `workspace?`
- `runtime`
- `interop?`
- `projectedImports`
- `deliveryBase`

The important thing is not the field names.
The important thing is that the model visibly distinguishes:
- truth
- derived truth
- startup augmentation
- interop residue

---

## C. Delivery handle
The launch seam should receive a minimal delivery object.
Conceptually something like:

- `kind`
- `path?`
- `dispose?`

The point is:
- wrangle receives a stable handle
- wrangle does not need to know projection internals
- bootstrap delivery internals can evolve without changing the command seam

## The final data flow
This is the final top-down flow the rewrite should target.

## 1. `Wrangle.command(...)` begins
It gathers:
- paths
- subcommand
- env/permission context
- Vite runtime facts

## 2. It asks `Bootstrap.Project` for a startup authority model
This projection is:
- ranked
- minimal
- truthful
- free of delivery mechanics

## 3. It asks `Bootstrap.Deliver` for a child-consumable handle
This step is:
- deterministic
- private to the driver
- stable across equivalent runs

## 4. Wrangle assembles the child command
With:
- the proven loader mode
- the existing runtime permissions/env
- the optional bootstrap handle

## 5. The child starts
At this point the bootstrap seam has completed its main job.

## 6. Upstream/runtime seams take over
- Vite native config loader
- `@deno/vite-plugin`
- app/plugin/transport resolution

This flow is the architecture.

## The decisive design choices
These are the final-shape decisions that matter most.

## 1. Projection over reconstruction
The final seam must project only startup-relevant truth.
It must not reconstruct a large synthetic compatibility world.

## 2. Ranked authority over flat merge
Consumer truth remains primary.
Derived workspace truth and startup additions remain visibly secondary.
Interop stays secondary.
Delivery stays last.

## 3. Stable delivery over disposable artifact churn
Equivalent startup authority must produce equivalent delivery identity.
If this is not true, the final shape has failed.

## 4. Driver-owned location over consumer-root leakage
Any remaining delivery artifact belongs in a driver-owned/private location.
Visible residue in package roots is not acceptable final architecture.

## 5. Narrow launch seam over broadened bootstrap power
Wrangle should launch the child, not absorb more bootstrap truth logic.
Bootstrap should enable startup, not grow into a runtime empire.

## 6. Early bridge over parallel universe
The final seam is an early bridge into the upstream-owned runtime world.
It is not a second Deno/Vite universe.

---

## Preferred endpoint vs acceptable fallback
The architecture should be robust enough to survive either of these implementation outcomes.

## Preferred endpoint
No generated artifact is needed.
Startup authority is handed to the child through a stable mechanism without visible file residue.

## Acceptable fallback endpoint
A generated artifact remains, but only if it is:
- deterministic or content-addressed
- private to the driver
- stable across equivalent runs
- conceptually subordinate to the projection model

### What is not acceptable
- random UUID identity
- cleanup-defined conceptual API
- consumer-root bootstrap debris
- synthetic file delivery treated as the model itself
