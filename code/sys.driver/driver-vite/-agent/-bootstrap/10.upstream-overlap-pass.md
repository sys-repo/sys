## Distilled question
Which parts duplicate or compensate for upstream tools, and which parts are truly owned here?

## Distilled answer
The current bootstrap seam is **not** mainly guilty of reimplementing all of Deno/Vite.
Its real shape is narrower and more specific:

**it is compensating for a hook-timing gap that upstream does not cover.**

That is the decisive overlap judgment.

So the current line should not be read as:
- “we built our own Deno resolver for fun”

It should be read as:
- “the child Vite 8 config/bootstrap path needs Deno-aware startup authority before the normal upstream resolution hooks exist, so `driver-vite` had to bridge that gap.”

That part is rightly owned here.

Where the current line goes too far is different:
- it over-owns some authority projection details
- it over-owns some delivery mechanics
- it duplicates some config/authority discovery patterns more broadly than necessary

So the real engineering conclusion is:

**the custom seam is justified, but its ownership envelope is currently too wide.**

That is the correct upstream-overlap read.

---

## The key overlap test
A very simple test makes this pass clear:

### If the concern exists only after Vite has entered its normal plugin/runtime pipeline
Then upstream should usually own it.

### If the concern exists because the child must start before those hooks are available
Then `driver-vite` likely owns it.

That is the main line.

Applied here:
- config/bootstrap-time authority bridging → owned here
- in-pipeline Deno module resolution → upstream/plugin-owned
- Vite config execution semantics → upstream/Vite-owned
- workspace alias/export projection for app/config composition → local `m.vite.config.workspace`-owned

This pass is mostly about keeping those boundaries honest.

## What `@deno/vite-plugin` already owns
From its README and implementation, `@deno/vite-plugin` already owns:

- locating Deno config for a Vite root
- creating Deno loaders
- environment-specific resolution caches
- `resolveId(...)` for Deno-aware module resolution inside Vite
- `load(...)` for Deno-loaded module content
- `npm:` / `jsr:` / `http:` / `https:` handling in the plugin pipeline
- some runtime specifier rewriting needed for Vite's module runner

This is real ownership.

It already carries the normal “inside Vite's resolution/load pipeline” Deno integration burden.

### Important limitation
Its README also says plainly:

- Deno-specific resolution cannot be used in `vite.config.ts`

That limitation is the whole reason your bootstrap seam exists.

So the right comparison is not:
- “why are we not just deleting everything in favor of `@deno/vite-plugin`?”

The right comparison is:
- “what must exist *because* `@deno/vite-plugin` stops at the config/bootstrap boundary, and what should still be left to it after that boundary?”

---

## What Vite native config loader already owns
Vite itself already owns:
- config execution semantics
- native config-loader behavior
- the actual config-evaluation path once the child is running in the acceptable mode

Your current line already respects this with:
- `--configLoader=native` for Vite 8+

That is a good upstream boundary.

### Correct read
`driver-vite` should not try to become its own config-loader implementation.
It should only make it possible for Vite's native config-loader path to start truthfully under Deno.

That means:
- delegating config execution to Vite is correct
- emulating or broadening Vite config-loader behavior locally would be over-ownership

---

## What Deno loader/workspace resolution already owns
Through Deno config/workspace/import-map semantics and, in upstream plugin land, `@deno/loader`, the upstream Deno side already owns:

- actual Deno import resolution semantics
- workspace-aware resolution behavior
- import-map interpretation
- remote/npm/jsr graph behavior
- loader-level module graph entrypoint handling

This means `driver-vite` should not aspire to become a second general-purpose Deno resolver.

### Important nuance
The current bootstrap seam does **not** fully reimplement Deno resolution.
It mostly does two earlier-stage things:
- discovers authority inputs
- fabricates a child-consumable authority handle

That is less bad than a full duplicate resolver.
But it is still overlap at the authority-discovery/projection layer.

---

## What `m.vite.config.workspace` already owns
This local module already owns a cleaner, narrower responsibility:

- deterministic alias projection from workspace/package exports
- explicit workspace-based config/app composition help

It is not a bootstrap seam.
It is not a child command seam.
It is not a Deno runtime loader.

It is a local positive model for:
- narrow truth inputs
- deterministic projection
- low-residue output

That matters because it shows `driver-vite` already contains a healthier pattern than the current bootstrap rescue shape.

## Overlap classification by surface

## 1. Overlap with `@deno/vite-plugin`
### Rightly owned here
- config/bootstrap-time Deno authority bridging before plugin hooks exist

### Upstream-owned
- ongoing Vite resolution/load behavior after startup
- Deno specifier handling within the Vite plugin pipeline

### Current mistake to avoid
Do not let bootstrap expand into app/plugin resolution just because the plugin does not cover `vite.config.ts`.
That would confuse a missing early hook with a missing full integration.

### Verdict
- **mostly compensation, not wrongful duplication**

---

## 2. Overlap with Vite native config loader
### Rightly owned here
- selecting the proven compatible launch mode
- supplying enough startup authority for the native loader to run truthfully

### Upstream-owned
- config execution semantics themselves

### Current mistake to avoid
Do not localize more config-loader logic than necessary.
The goal is to feed Vite's loader, not replace it.

### Verdict
- **good delegation currently present**
- keep it that way

---

## 3. Overlap with Deno loader/workspace resolution
### Rightly owned here
- collecting startup-relevant Deno authority from consumer/workspace state
- projecting that authority into a child-startup handle

### Upstream-owned
- the actual resolution semantics once the child runtime is operating in Deno-aware mode

### Current overreach
The current bootstrap line broadens local ownership by:
- merging root + nearest authority too broadly
- carrying more import-map JSON shape than the child contract has clearly earned
- injecting synthetic authority delivery state into the operational center

This is not because `driver-vite` must own Deno resolution.
It is because the current compensation mechanism has become too broad.

### Verdict
- **justified overlap at the projection boundary**
- **too much breadth inside that boundary**

## 4. Overlap with `m.vite.config.workspace`
### Shared source truth
Both seams inspect workspace/package truth.

### Different outputs
- `m.vite.config.workspace` → aliases for config/app composition
- bootstrap seam → child startup authority bridge

### Important read
This is **not** direct responsibility duplication.
It is source-level overlap with different products.

That said, bootstrap should learn from this module's shape:
- deterministic
- narrow
- projection-oriented

### Verdict
- **not redundant**
- **but a strong design reference**

## What `driver-vite` is rightly owning
This is the ownership core that should survive the rewrite.

## 1. The pre-plugin bootstrap gap
This is the big one.
Upstream does not own the moment before `vite.config.ts` can be evaluated under the right Deno-aware authority.
That is your seam.

## 2. Child launch shaping under Deno
This includes:
- command assembly
- permissions
- env wiring
- choosing the correct config-loader mode
- threading optional bootstrap delivery into the child invocation

This is integration-driver work.
It is rightly local.

## 3. Narrow startup authority projection
`driver-vite` should own the minimum truthful projection needed for the child to start.
Not more.
But that minimum is real local ownership.

---

## What `driver-vite` is compensating for
These are the things that are not ideal local ownership, but are currently understandable because of upstream/tooling seams.

## 1. Synthetic bootstrap delivery
Using a generated import-map artifact is a compensation mechanism for the lack of an earlier cleaner hook.
It is not intrinsically the model.

## 2. Bootstrap-owned Vite runtime dependency projection
The hard-coded runtime imports and subpaths are compensating for the fact that the child must see enough toolchain authority before ordinary Vite/plugin resolution can help.

## 3. Some package-metadata interop checks
These are compensations for real npm/Vite/Deno boundary friction.
But they should remain sharply scoped and secondary.

---

## What looks unnecessarily reimplemented or over-owned
This is the part the rewrite should challenge hardest.

## 1. Broad authority reconstruction
The current line reconstructs a larger synthetic startup world than the essential contract has earned.
That is over-ownership.

## 2. Whole-document import-map carry-through
Carrying broad JSON shape forward is much closer to re-hosting config authority than narrowly projecting child startup needs.
That is likely more than this seam should own.

## 3. Multiple local config/authority discovery shapes
Today there are several nearby ways of discovering related truth:
- bootstrap source discovery
- `@deno/vite-plugin` config discovery
- `m.vite.config.workspace` workspace discovery

Not all of this is wrong.
But it suggests a future narrower shared discovery primitive may exist beneath the seams, even if the seams themselves remain distinct.

## 4. Synthetic delivery mechanics defining the seam conceptually
Random artifact names, cleanup lifecycle, and file-URL injection are delivery mechanics.
When they become the center of the seam story, local ownership has drifted too far from the real contract.

---

## The deepest upstream-overlap insight
The deepest insight of this pass is:

**the current bootstrap line is compensating for an upstream timing hole, not for a missing upstream resolution engine.**

That distinction matters enormously.

Because if you misread the problem as:
- “upstream cannot do Deno resolution, so we need our own resolution world”

you will overbuild.

But if you read it correctly as:
- “upstream cannot apply its normal resolution help early enough for `vite.config.ts` bootstrap”

you get the right target:
- a narrow early-bridge seam
- not a parallel resolution stack

That is the main architectural correction in this pass.

---

## Consequences for the replacement
This pass implies the rewrite should:

### 1. Preserve the custom bootstrap seam
Because upstream does not fully cover the decisive early boundary.

### 2. Narrow the seam aggressively
Because the seam should compensate only for hook timing, not broadly own Deno/Vite resolution.

### 3. Delegate downstream resolution back to upstream as soon as possible
Once the child is alive in the right authority world, let Vite native loader and `@deno/vite-plugin` do their jobs.

### 4. Keep workspace alias/export projection separate
`m.vite.config.workspace` is not redundant with bootstrap; it is a separate local authority-projection seam.

### 5. Treat shared discovery logic as a possible helper, not a reason to collapse seams
A future shared primitive may be good.
A future merged mega-seam would not.

### 6. Constrain what counts as a legitimate bridge
A bridge is not justified merely because it runs early.
It is justified only if it remains:
- startup-only
- projection/delivery-only
- non-governing after handoff
- unable to absorb app/plugin/transport/runtime concerns without becoming architectural debt

That constraint is part of the doctrine, not an optional later refinement.

---

## What not to do after this pass

### 1. Do not delete bootstrap because `@deno/vite-plugin` exists
Its README explicitly rules out the decisive boundary.

### 2. Do not let bootstrap absorb more of the plugin/runtime world
That would be solving the wrong problem.

A seam is already too wide if it:
- owns behavior after startup handoff
- compensates for app/plugin/transport/runtime concerns
- requires broad payload carry-through to stay green
- cannot still be described truthfully as startup authority projection + delivery

### 3. Do not assume all overlap is bad
Some overlap is legitimate because the seams happen at different phases.

### 4. Do not confuse shared source truth with duplicated responsibility
Two seams can read the same workspace truth and still have different legitimate outputs.

---

## Distilled engineering conclusion
The custom bootstrap seam is justified.
That is the first and most important conclusion of this pass.

But it is justified for a very specific reason:
- it bridges an upstream timing gap at the config/bootstrap boundary

It is **not** justified as a broad parallel reimplementation of Deno/Vite resolution.
And whenever the current line drifts toward that broader shape, it becomes architectural debt.

So the clean summary is:

**own the early bridge, delegate the rest.**

But that sentence is only safe if “bridge” is read narrowly and concretely:
- startup-only
- projection/delivery-only
- handoff-early
- never a cover story for broad bootstrap ownership

That is the correct upstream-overlap doctrine for the rewrite.
